.Strategische Rationale
****
**Waarom deze libraries?**

1. **Pattern Recognition**: WBA is representatief voor B2B/enterprise applicaties die deze functionaliteit standaard nodig hebben
2. **Competitive Advantage**: Django en Rails hebben deze patterns al als packages - Boundary zou "batteries included" moeten zijn
3. **ROI Multiplier**: Elke library bespaart €3k-€12k per project en is herbruikbaar over tientallen projecten
4. **Reduced Vendor Lock-in**: Eigen search/workflow libraries elimineren afhankelijkheden van Elasticsearch, workflow-as-a-service providers, etc.
****

== Inleiding

=== Context: WBA Conversie Analyse

De analyse van de Django Werkbezoek Application (WBA) naar Boundary Framework heeft belangrijke inzichten opgeleverd over functionaliteit die Boundary momenteel *niet* biedt, maar die *wel* essentieel is voor enterprise applicaties.

.WBA Applicatie Kenmerken
* **Domein**: Event management voor culturele werkbezoeken
* **Codebase**: ~80k lines Python, 95 templates, 15+ Django apps
* **Complexiteit**: State machines, workflows, search, geo, PDF generatie, scheduling
* **Typisch voor**: B2B/enterprise applicaties met goedkeuringsprocessen en rapportages

=== Methodologie

Dit document identificeert strategische gaps door:

1. **Pattern Analysis**: Welke functionaliteit in WBA is niet-triviaal en veelvoorkomend?
2. **Abstraction Assessment**: Welke patterns zijn generiek genoeg voor herbruikbare libraries?
3. **ROI Calculation**: Hoeveel development tijd bespaart een library per project?
4. **Ecosystem Comparison**: Wat bieden Django/Rails/Laravel wel dat Boundary niet heeft?
5. **Priority Ranking**: Wat heeft de hoogste impact vs effort ratio?

=== Scope

Dit document richt zich op **framework-level libraries** die:

* ✅ Herbruikbaar zijn over 5+ projecten
* ✅ Non-triviale abstracties bieden (niet simpele wrappers)
* ✅ Common enterprise patterns adresseren
* ✅ Boundary's FC/IS architectuur respecteren
* ❌ NIET niche-functionaliteit (te specifiek voor 1-2 projecten)
* ❌ NIET externe service wrappers (die horen in `boundary-external`)

== Library 1: boundary-workflow

[discrete]
=== Priority: 🔥 **HOOG (1)** | Impact: **High** | ROI: **€8.000 - €12.000 per project**

=== Probleem Beschrijving

==== Wat WBA Heeft (Custom Implementation)

WBA's `workvisit` app bevat complexe state machine logic:

[source,python]
----
# Django FSM (Finite State Machine)
class WorkVisit(models.Model):
    state = FSMField(default='draft', protected=True)
    
    @fsm_log_by  # <1>
    @transition(field=state, source='draft', target='published',
                permission=lambda instance, user: user.has_perm('workvisit.publish'))
    def publish(self):
        self.published_at = timezone.now()
        send_notification_email(self)  # <2>
        notify_subscribers(self)
    
    @transition(field=state, source='published', target='archived',
                conditions=[is_past_end_date])  # <3>
    def archive(self):
        clear_cache(self)
----
<1> Automatische audit trail van state changes
<2> Side effects bij transitions (emails, notificaties)
<3> Conditional transitions (alleen als event afgelopen is)

**Complexiteit in WBA**:

* ~1500 lines state machine logic in `workvisit/models/workvisit.py`
* Permissions integratie (wie mag welke transitions uitvoeren?)
* Audit logging (wie heeft wanneer status gewijzigd?)
* Side effects orchestration (emails, cache invalidation, notificaties)
* Conditional transitions (alleen als data validaties slagen)

==== Waarom Dit Een Strategische Gap Is

[cols="1,3"]
|===
|Aspect |Rationale

|**Ubiquity**
|80% van B2B apps heeft workflows: orders (pending→paid→shipped), tickets (open→in-progress→resolved), documents (draft→review→approved)

|**Complexity**
|State machines zijn *moeilijk goed te doen*: race conditions, audit trails, permissions, rollbacks

|**Boilerplate**
|Zonder library schrijf je per project 500-1500 lines boilerplate voor FSM logic

|**Vendor Lock-in**
|Zonder eigen library grijpen teams naar externe services (Temporal.io, Workflow86) met maandelijkse kosten + vendor lock-in
|===

=== Voorgestelde Oplossing: boundary-workflow

==== Library Design

**Core Abstractions**:

[source,clojure]
----
(ns boundary.workflow.core)

;; Declaratieve workflow definitie
(defworkflow workvisit-workflow
  {:entity :workvisit
   
   :states {:draft      {:label "Concept"
                         :color :gray}
            :published  {:label "Gepubliceerd"
                         :color :green}
            :archived   {:label "Gearchiveerd"
                         :color :blue}
            :deleted    {:label "Verwijderd"
                         :color :red}}
   
   :transitions [{:id :publish
                  :from :draft
                  :to :published
                  :label "Publiceren"
                  :when [:user/has-permission? :workvisit/publish]  ; <1>
                  :validate [validate-required-fields              ; <2>
                             validate-future-date]
                  :actions [:notify-subscribers                     ; <3>
                            :send-email
                            :invalidate-cache]}
                 
                 {:id :archive
                  :from :published
                  :to :archived
                  :label "Archiveren"
                  :when [:workvisit/past-end-date?]
                  :auto? true}]  ; <4>
   
   :hooks {:on-enter-published [send-notification-fn
                                 update-published-at]
           :on-exit-draft [clear-draft-data]
           :on-any-transition [log-audit-trail]}})  ; <5>
----
<1> Permission checks (integreert met boundary-user)
<2> Data validatie voordat transition toegestaan wordt
<3> Declaratieve side effects (emails, cache, notificaties)
<4> Auto-transitions (cron job controleert conditions, voert uit)
<5> Lifecycle hooks voor custom logic

**Usage in Application Code**:

[source,clojure]
----
(ns myapp.workvisit.shell.service
  (:require [boundary.workflow.core :as wf]))

;; Transition uitvoeren
(defn publish-workvisit [workvisit-id user-id]
  (wf/transition! workvisit-workflow
                  {:entity-id workvisit-id
                   :transition :publish
                   :actor-id user-id
                   :context {:ip-address "192.168.1.1"}}))

;; => {:success? true
;;     :entity {:id uuid :state :published ...}
;;     :audit-entry {:transition :publish :from :draft :to :published
;;                   :actor-id uuid :timestamp inst}}

;; Of bij failure:
;; => {:success? false
;;     :error {:type :permission-denied
;;             :message "User lacks workvisit/publish permission"}}

;; Ophalen van mogelijke transitions voor entity
(wf/available-transitions workvisit-workflow workvisit user)
;; => [{:id :publish :label "Publiceren" :enabled? true}
;;     {:id :archive :label "Archiveren" :enabled? false
;;      :reason "Event is not past end date"}]
----

== Library 2: boundary-search

=== Probleem Beschrijving

==== Wat WBA Heeft (Elasticsearch Overkill)

WBA gebruikt **Elasticsearch 9.x** voor basic search functionaliteit:

[source,python]
----
# Django Elasticsearch DSL document definition
from django_elasticsearch_dsl import Document, fields

@registry.register_document
class WorkVisitDocument(Document):
    event_name = fields.TextField(
        analyzer='dutch',  # <1>
        fields={'raw': fields.KeywordField()}
    )
    event_city = fields.TextField(analyzer='dutch')
    description = fields.TextField(analyzer='dutch')
    
    class Index:
        name = 'workvisits'
        settings = {
            'number_of_shards': 1,
            'number_of_replicas': 0
        }

# Search view
class WorkVisitSearchView(APIView):
    def get(self, request):
        q = request.GET.get('q', '')
        search = WorkVisitDocument.search().query(
            "multi_match",
            query=q,
            fields=['event_name^3', 'event_city^2', 'description'],  # <2>
            type='best_fields',
            operator='and'
        )
        return Response(search[:20].execute().to_dict())
----
<1> Nederlandse stemming (zoeken op "muziek" vindt ook "muzikaal")
<2> Weighted field boosting (event_name 3x belangrijker dan description)

**Wat Elasticsearch WEL biedt (maar WBA NIET gebruikt)**:

* ❌ Faceted search (filters op categorieën)
* ❌ Geo-spatial queries (zoeken binnen radius)
* ❌ Aggregations (statistieken berekenen)
* ❌ Complex analytics
* ❌ Machine learning features

**Wat Elasticsearch KOST**:

* €3.600 - €7.200/jaar aan infrastructure (managed Elasticsearch cluster)
* Deployment complexity (extra service om te runnen/monitoren)
* Sync issues (database vs search index out-of-sync bugs)
* Development overhead (index management, reindexing scripts)

==== PostgreSQL FTS als Alternatief

PostgreSQL heeft **ingebouwde full-text search** die voor 80% van use cases voldoende is:

[source,sql]
----
-- GIN index voor snelle search
CREATE INDEX workvisit_search_idx ON workvisits 
USING GIN(to_tsvector('dutch', 
                      event_name || ' ' || event_city || ' ' || description));

-- Search query met ranking
SELECT 
  *,
  ts_rank_cd(
    to_tsvector('dutch', event_name || ' ' || event_city || ' ' || description),
    plainto_tsquery('dutch', 'muziek Amsterdam'),
    1|2|4|8  -- A=1.0, B=0.4, C=0.2, D=0.1 weights <1>
  ) as rank
FROM workvisits
WHERE to_tsvector('dutch', event_name || ' ' || event_city || ' ' || description)
      @@ plainto_tsquery('dutch', 'muziek Amsterdam')
ORDER BY rank DESC
LIMIT 20;
----
<1> Gewogen ranking: zelfde als Elasticsearch field boosting

**Performance**: <50ms voor 10.000 records (vs Elasticsearch 20-50ms, maar zonder extra infra)

=== Voorgestelde Oplossing: boundary-search

==== Library Design

**Core Abstractions**:

[source,clojure]
----
(ns boundary.search.core)

;; Declaratieve search index definitie
(defsearch workvisit-search
  {:table :workvisits
   :language :dutch  ; PostgreSQL text search language <1>
   
   :fields {:event-name {:weight :A}      ; ts_rank weight A (1.0)
            :event-city {:weight :B}      ; weight B (0.4)
            :description {:weight :C}     ; weight C (0.2)
            :tags {:weight :D}}           ; weight D (0.1)
   
   :filters [:tenant-id :status :event-type]  ; <2>
   
   :ranking :bm25  ; of :classic (default PostgreSQL) <3>
   
   :suggest? true  ; Enable typo suggestions <4>
   
   :highlight? true})  ; Highlight matches in results <5>
----
<1> Ondersteunt alle PostgreSQL text search languages (dutch, english, german, french, etc.)
<2> Filterable fields (WHERE clauses naast FTS)
<3> Ranking algoritme (BM25 is state-of-the-art, PostgreSQL default is klassiek TF-IDF)
<4> Optioneel: typo suggestions via trigram similarity
<5> Optioneel: highlight matching woorden in resultaten

**Usage**:

[source,clojure]
----
(ns myapp.workvisit.shell.service
  (:require [boundary.search.core :as search]))

;; Simpele search
(search/query workvisit-search 
              "muziek Amsterdam"
              {:limit 20})
;; => [{:id uuid :event-name "Jazz Concert" :rank 0.95 ...}
;;     {:id uuid :event-name "Classical Music Night" :rank 0.87 ...}]

;; Search met filters
(search/query workvisit-search
              "concert"
              {:filters {:status :published
                         :event-type :music}
               :limit 20
               :offset 0})

;; Suggesties voor typo's
(search/suggest workvisit-search "concrt")
;; => ["concert" "concerts"]

;; Highlighted results
(search/query workvisit-search
              "Amsterdam"
              {:highlight? true})
;; => [{:event-city "<b>Amsterdam</b>" ...}]
----

==== Migration Path van Elasticsearch

Voor teams die *wel* Elasticsearch gebruiken, biedt de library een compatibele API:

[source,clojure]
----
;; resources/conf/prod/config.edn
{:boundary/search-service
 {:backend :elasticsearch  ; of :postgresql (default)
  :elasticsearch-url "https://elasticsearch.prod:9200"
  :index-prefix "myapp-"}}

;; Code blijft identiek - library switcht backend transparant
(search/query workvisit-search "query" {...})
----

**Use case**: Teams kunnen starten met PostgreSQL FTS, later upgraden naar Elasticsearch als ze 1M+ records hebben of advanced features nodig hebben.

==== Database Schema & Indexing

[source,sql]
----
-- Generated tsvector column voor performance
ALTER TABLE workvisits 
ADD COLUMN search_vector tsvector 
GENERATED ALWAYS AS (
  setweight(to_tsvector('dutch', coalesce(event_name, '')), 'A') ||
  setweight(to_tsvector('dutch', coalesce(event_city, '')), 'B') ||
  setweight(to_tsvector('dutch', coalesce(description, '')), 'C') ||
  setweight(to_tsvector('dutch', coalesce(array_to_string(tags, ' '), '')), 'D')
) STORED;

-- GIN index (fast search, slower writes)
CREATE INDEX workvisits_search_idx ON workvisits USING GIN(search_vector);

-- Optioneel: trigram index voor typo suggestions
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX workvisits_event_name_trgm_idx ON workvisits 
USING GIN(event_name gin_trgm_ops);
----

==== Wat WBA Heeft (Custom PDF Generation)

WBA heeft een `pdfs` app voor rapportages:

[source,python]
----
# Django view voor PDF generatie
from reportlab.lib.pagesizes import A4, landscape
from reportlab.platypus import SimpleDocTemplate, Table, TableStyle

def workvisit_overview_pdf(request):
    response = HttpResponse(content_type='application/pdf')
    response['Content-Disposition'] = 'attachment; filename="workvisits.pdf"'
    
    doc = SimpleDocTemplate(response, pagesize=landscape(A4))
    elements = []
    
    # Header
    elements.append(Paragraph("Werkbezoeken Overzicht", styles['Title']))
    
    # Tabel met data
    data = [['Event', 'Datum', 'Stad', 'Deelnemers']]
    for visit in WorkVisit.objects.filter(status='published'):
        data.append([visit.event_name, visit.event_date, 
                     visit.event_city, visit.attendees_count])
    
    table = Table(data)
    table.setStyle(TableStyle([
        ('BACKGROUND', (0, 0), (-1, 0), colors.grey),
        ('GRID', (0, 0), (-1, -1), 1, colors.black)
    ]))
    elements.append(table)
    
    doc.build(elements)
    return response
----

**Complexiteit**:

* ~500-1000 lines boilerplate code voor PDF layouts
* Excel exports apart geïmplementeerd (openpyxl library)
* Styling hardcoded in Python (geen templates)
* Geen herbruikbare componenten

[source,python]

# Django model
class Event(models.Model):
    title = models.CharField(max_length=200)
    start_date = models.DateTimeField()
    end_date = models.DateTimeField()
    recurrence_rule = models.CharField(max_length=500, blank=True)  # iCal RRULE
    timezone = models.CharField(max_length=50, default='Europe/Amsterdam')
    
    def get_occurrences(self, start, end):
        """Bereken alle occurrences van recurring event tussen start/end."""
        if not self.recurrence_rule:
            return [self] if start <= self.start_date <= end else []
        
        # Parse RRULE en genereer occurrences
        rrule = dateutil.rrule.rrulestr(self.recurrence_rule, 
                                         dtstart=self.start_date)
        return list(rrule.between(start, end))
    
    def conflicts_with(self, other_event):
        """Check of events overlappen in tijd."""
        return (self.start_date < other_event.end_date and
                self.end_date > other_event.start_date)
----

import googlemaps

def geocode_address(postcode, city):
    """Zet postcode+stad om naar lat/long via Google Geocoding API."""
    gmaps = googlemaps.Client(key=settings.GOOGLE_MAPS_API_KEY)
    result = gmaps.geocode(f"{postcode}, {city}, Netherlands")
    
    if result:
        location = result[0]['geometry']['location']
        return location['lat'], location['lng']
    return None, None

def reverse_geocode(lat, lng):
    """Zet lat/long om naar adres."""
    gmaps = googlemaps.Client(key=settings.GOOGLE_MAPS_API_KEY)
    result = gmaps.reverse_geocode((lat, lng))
    
    if result:
        return result[0]['formatted_address']
    return None

==== Library Design

**Core Abstractions**:

[source,clojure]

(ns boundary.geo.core)

;; Config in system.edn
{:boundary/geo-service
 {:provider :google  ; of :openstreetmap, :mapbox, :azure-maps
  :api-key #env BND_GEO_API_KEY

              {:lat 52.092 :lng 5.104}) ; Utrecht
;; => 42.3  ; kilometers

==== Multi-Provider Support

**Waarom multiple providers?**

* **Google Maps**: Best accuracy, maar duur (€5-€40 per 1000 requests)
* **OpenStreetMap (Nominatim)**: Gratis, maar rate limits + minder accuraat
* **Mapbox**: Goede prijs/kwaliteit, €0.50 per 1000 requests
* **Azure Maps**: Included in Azure subscriptions (vaak gratis voor bestaande klanten)

Library abstraheert verschillen:

[source,clojure]
----
;; Switch provider via config - code blijft identiek
{:boundary/geo-service
 {:provider :openstreetmap  ; gratis voor low-volume apps
  :cache-ttl 86400}}

;; Of fallback chain (probeer gratis eerst, fallback naar betaald)
{:boundary/geo-service
 {:providers [:openstreetmap :google]
  :openstreetmap-rate-limit 1
  :google-api-key #env GOOGLE_API_KEY}}
----

* **WBA met libraries**: 2.5 + 1 + 1 dagen = **4.5 dagen overhead**

**Savings**: 21.5-35.5 dagen × €800/dag = **€17.2k - €28.4k per project**

==== 2. **Lower Vendor Lock-in Risks**

Zonder eigen libraries grijpen teams naar externe services:

* **Workflow**: Temporal.io (€500-€2000/maand), Workflow86 (€199-€999/maand)
* **Search**: Algolia (€1/1000 searches), Elasticsearch Cloud (€300-€600/maand)
* **Reports**: DocRaptor (€50-€500/maand), CloudConvert (€9-€99/maand)

**Costs**: €1000-€4000/maand × 12 = **€12k-€48k/jaar per project**

Met eigen libraries: **€0 operational costs** (behalve compute, wat je toch al betaalt).

==== 3. **Consistency Across Projects**

Zonder libraries heeft elk project:

* Andere workflow implementation (custom FSM code)
* Andere search provider (Elasticsearch vs PostgreSQL FTS vs Algolia)
* Andere report templates (elke dev maakt eigen PDF layouts)

**Met libraries**:

* ✅ Uniform API across projects
* ✅ Shared patterns (nieuwe devs herkennen code)
* ✅ Easier maintenance (bugfixes in library benefits all projects)

==== 4. **Competitive Advantage in Sales**

Bij enterprise RFPs wordt vaak gevraagd:

* "Heeft jullie framework workflow/approval support?" → **JA** (boundary-workflow)
* "Kunnen we full-text search zonder Elasticsearch?" → **JA** (boundary-search)
* "Kan het facturen/rapporten genereren?" → **JA** (boundary-reports)
* "Ondersteuning voor recurring events?" → **JA** (boundary-calendar)

Zonder deze libraries: **NEEN** (of "ja maar custom development, +€20k kosten")

=== boundary-workflow: Complete Example

[source,clojure]
----
(ns myapp.order.workflow
  (:require [boundary.workflow.core :as wf]))

;; Order workflow definitie
(defworkflow order-workflow
  {:entity :order
   
   :states {:pending    {:label "Pending Payment" :color :yellow}
            :paid       {:label "Paid" :color :green}
            :shipped    {:label "Shipped" :color :blue}
            :delivered  {:label "Delivered" :color :green}
            :cancelled  {:label "Cancelled" :color :red}}
   
   :transitions [{:id :pay
                  :from :pending
                  :to :paid
                  :label "Mark as Paid"
                  :when [:user/has-permission? :order/process]
                  :validate [validate-payment-received]
                  :actions [:send-confirmation-email
                            :notify-warehouse
                            :create-invoice]}
                 
                 {:id :ship
                  :from :paid
                  :to :shipped
                  :label "Ship Order"
                  :when [:user/has-permission? :order/ship]
                  :validate [validate-inventory-available]
                  :actions [:generate-shipping-label
                            :update-inventory
                            :send-tracking-email]}
                 
                 {:id :deliver
                  :from :shipped
                  :to :delivered
                  :label "Mark as Delivered"
                  :auto? true  ; Auto-transition when tracking shows delivered
                  :when [:order/tracking-delivered?]
                  :actions [:send-delivery-confirmation
                            :request-review]}
                 
                 {:id :cancel
                  :from [:pending :paid]  ; Multiple source states
                  :to :cancelled
                  :label "Cancel Order"
                  :when [:user/has-permission? :order/cancel]
                  :actions [:refund-payment
                            :send-cancellation-email
                            :restore-inventory]}]
   
   :hooks {:on-enter-paid [create-invoice-fn
                           notify-accounting-fn]
           :on-enter-shipped [start-tracking-monitor-fn]
           :on-any-transition [log-audit-trail-fn]}})

;; Usage in service layer
(defn process-payment [order-id payment-data user-id]
  (let [result (wf/transition! order-workflow
                                {:entity-id order-id
                                 :transition :pay
                                 :actor-id user-id
                                 :context {:payment-data payment-data}})]
    (if (:success? result)
      {:status 200 :body {:message "Payment processed"
                          :order (:entity result)}}
      {:status 400 :body {:error (:error result)}})))

;; Admin UI integration
(admin/entity-config :orders
  {:detail-view {:workflow order-workflow  ; Show state + available transitions
                 :show-audit-log? true}})
----

=== boundary-search: Complete Example

[source,clojure]
----
(ns myapp.product.search
  (:require [boundary.search.core :as search]))

;; Product search index
(defsearch product-search
  {:table :products
   :language :dutch
   
   :fields {:name {:weight :A}          ; Hoogste priority
            :brand {:weight :A}
            :description {:weight :B}
            :tags {:weight :C}
            :sku {:weight :D}}          ; Laagste priority
   
   :filters [:category :price-min :price-max :in-stock? :tenant-id]
   
   :ranking :bm25
   :suggest? true
   :highlight? true})

;; HTTP endpoint voor product search
(defn search-products-handler [request]
  (let [query (get-in request [:params :q])
        filters (select-keys (:params request) [:category :price-min :price-max])
        results (search/query product-search query
                              {:filters filters
                               :limit 20
                               :offset (parse-int (get-in request [:params :page] "0"))
                               :highlight? true})]
    {:status 200
     :body {:results results
            :total (search/count product-search query {:filters filters})}}))

;; Autocomplete suggestions
(defn autocomplete-handler [request]
  (let [partial (get-in request [:params :q])]
    {:status 200
     :body {:suggestions (search/suggest product-search partial)}}))

;; Admin UI integration (search bar in entity list)
(admin/entity-config :products
  {:list-view {:search-index product-search  ; Enables search bar
               :search-fields [:name :brand :sku]}})

=== Van Custom FSM naar boundary-workflow

[source,clojure]
----
;; Before (custom state machine)
(defn publish-workvisit [workvisit user]
  (when-not (= (:state workvisit) :draft)
    (throw (ex-info "Can only publish drafts" {:type :invalid-state})))
  (when-not (user/has-permission? user :workvisit/publish)
    (throw (ex-info "No permission" {:type :permission-denied})))
  
  (let [updated (assoc workvisit 
                       :state :published
                       :published-at (now))]
    (db/update! :workvisits (:id workvisit) updated)
    (send-notification-email updated)
    (invalidate-cache (:id workvisit))
    updated))

;; After (boundary-workflow)
(defworkflow workvisit-workflow
  {:entity :workvisit
   :states {:draft {...} :published {...}}
   :transitions [{:id :publish
                  :from :draft
                  :to :published
                  :when [:user/has-permission? :workvisit/publish]
                  :actions [:send-notification-email :invalidate-cache]}]})

(defn publish-workvisit [workvisit-id user-id]
  (wf/transition! workvisit-workflow
                  {:entity-id workvisit-id
                   :transition :publish
                   :actor-id user-id}))

;; 15-20 lines → 5 lines, declarative, audit trail, permission checks built-in
----

=== Library Categorisatie

==== Must-Have (Priority 1-2) 🔥

**boundary-workflow + boundary-search**

* **Impact**: Hoogste ROI (€7k-€12k per project)
* **Frequency**: 80% van B2B projecten heeft deze nodig
* **Competition**: Django/Rails hebben deze al - table stakes
* **Risk**: Hoog risico om klanten te verliezen zonder deze features

==== Should-Have (Priority 3-4, 6-7) 🟡

**boundary-reports + boundary-calendar + boundary-push + boundary-audience**

* **Impact**: Medium ROI (€3k-€6k per project)
* **Frequency**: 40-60% van projecten heeft deze nodig
* **Competition**: Standaard in Django/Rails ecosysteem
* **Risk**: Medium risico - kan met workarounds maar kost tijd

==== Nice-to-Have (Priority 5, 8) 🟢

**boundary-geo + boundary-forms**

* **Impact**: Laag ROI (€400-€4k per project)
* **Frequency**: 20-40% van projecten heeft deze nodig
* **Competition**: Beschikbaar maar niet standaard
* **Risk**: Laag risico - third-party alternatieven beschikbaar

== Phase 1: High-Impact B2B Foundations

**Timeline**: Q2 2026 (6.5 weken) +
**Investment**: €26.400 (33 development days) +
**Focus**: Solve 80% van B2B pain points

=== Libraries

==== 1. boundary-workflow

[cols="1,3"]
|===
|Effort |18 dagen (€14.400)
|ROI per project |€8.000 - €12.000
|Breakeven |2 projecten
|Frequency |80% van B2B apps
|===

**Wat het doet**:

* Declaratieve state machine definitie
* Permission-based transitions
* Automatische audit trails
* Side effects orchestration (emails, cache, notificaties)
* Auto-transitions (cron-based)

**Integraties**:

* `boundary-user` → Permission checks
* `boundary-jobs` → Auto-transitions
* `boundary-admin` → Workflow visualizer

**Voorbeelden**:

* E-commerce: Order workflow (pending → paid → shipped → delivered)
* CRM: Lead workflow (new → contacted → qualified → converted)
* Ticketing: Ticket workflow (open → in-progress → resolved → closed)
* Document Management: Document workflow (draft → review → approved → published)

**Success Criteria**:

* [x] Workflow definitie via `defworkflow` macro
* [x] Transitions met permissions + validations
* [x] Audit log met actor, timestamp, context
* [x] Admin UI: workflow visualizer + audit trail viewer
* [x] Integration tests met boundary-user, boundary-jobs
* [x] AGENTS.md met voorbeelden

==== 2. boundary-search

[cols="1,3"]
|===
|Effort |15 dagen (€12.000)
|ROI per project |€7.000 - €10.000 + €3.6k-€7.2k/jaar opex savings
|Breakeven |1.5 projecten
|Frequency |70% van apps
|===

**Wat het doet**:

* PostgreSQL Full-Text Search wrapper
* Multi-language stemming (dutch, english, german, french, etc.)
* Weighted field boosting
* Typo suggestions (trigram similarity)
* Result highlighting
* Optional Elasticsearch adapter (migration path)

**Waarom belangrijk**:

* **Elasticsearch is overkill** voor 80% use cases
* WBA gebruikt Elasticsearch alleen voor basic search (geen facets, geo, ML)
* **Cost savings**: €3.6k-€7.2k/jaar aan Elasticsearch infrastructure
* **Simplicity**: Geen sync issues, geen extra service om te runnen

**Integraties**:

* `boundary-platform` → Database context
* `boundary-cache` → Query result caching
* `boundary-admin` → Search index management

**Voorbeelden**:

* Content platforms: Zoeken in artikelen, blogs
* E-commerce: Product zoeken (naam, beschrijving, tags)
* Document management: Zoeken in documenten
* CRM: Zoeken in contacten, bedrijven, notities

**Success Criteria**:

* [x] Search definitie via `defsearch` macro
* [x] PostgreSQL FTS met GIN indexing
* [x] Multi-language support (10+ languages)
* [x] Typo suggestions + highlighting
* [x] Elasticsearch adapter (fallback voor >1M records)
* [x] Performance benchmarks (<50ms voor 10k records)
* [x] AGENTS.md met migration guide

=== Rationale

2. **Snelste breakeven**: 1.5-2 projecten

5. **Eliminates vendor lock-in**: Vervangt Temporal.io (€500-€2000/maand) en Elasticsearch (€300-€600/maand)

**Risk**: Laag - patterns zijn bewezen en goed begrepen.

**Timeline**: Q4 2026 (7.5 weken) +
**Investment**: €31.200 (39 development days) +

=== Libraries

==== 6. boundary-push

[cols="1,3"]
|===
|Effort |21 dagen (€16.800)
|ROI per project |€4.000 - €6.000
|Breakeven |3-4 projecten
|Frequency |50% van projecten (met mobile apps)
|===

**Wat het doet**:

* Multi-platform push notifications (Firebase/APNs)
* Device token management (registration, rotation, cleanup)
* Scheduled pushes (via boundary-jobs)
* Deep linking (navigate naar specifieke screens)
* Multi-language support
* Analytics (delivery rates, open rates)
* Silent notifications (data-only)

**Waarom belangrijk**:

* **Mobile-first**: Push notifications zijn critical voor mobile app engagement
* **Retention**: 3-10x hogere retention dan email-only
* **Self-hosted**: Eliminates OneSignal/Pusher (€0-€999/maand)

**Integraties**:

* `boundary-user` → Device tokens per user
* `boundary-jobs` → Scheduled pushes
* `boundary-audience` → Targeted pushes
* `boundary-observability` → Analytics logging

**Voorbeelden**:

* E-commerce: Order updates, promo notifications, cart abandonment
* Social apps: New followers, messages, likes
* Fintech: Transaction alerts, balance updates
* Healthcare: Appointment reminders, medication alerts
* On-demand services: Driver arrival, order status (Uber, DoorDash)

**Success Criteria**:

* [ ] Push definitie via `defpush` macro
* [ ] Firebase adapter (FCM SDK wrapper)
* [ ] APNs adapter (HTTP/2 wrapper)
* [ ] Device token management (registration, rotation)
* [ ] Scheduled pushes via boundary-jobs
* [ ] Deep linking support
* [ ] Analytics (status tracking, delivery/open rates)
* [ ] AGENTS.md met Firebase/APNs setup

==== 7. boundary-audience

[cols="1,3"]
|===
|Effort |18 dagen (€14.400)
|ROI per project |€3.000 - €5.000
|Breakeven |4 projecten
|Frequency |40% van projecten
|===

**Wat het doet**:

* Complex user segmentation/targeting
* Declaratieve filter builder (age, location, behavior, tenure, etc.)
* Dynamic audience building (runtime)
* Caching & performance optimization
* Admin UI audience builder

**Waarom belangrijk**:

* **Reusable pattern**: Audience targeting wordt gebruikt door push, email, campaigns, forms
* **Engagement**: Targeted communications hebben 2-5x hogere engagement
* **Eliminates duplication**: OTTO had 8 verschillende filter types herhaald over Notification + Survey models

**Integraties**:

* `boundary-push` → Targeted push notifications
* `boundary-email` → Email campaigns
* `boundary-forms` → Targeted surveys
* `boundary-jobs` → Scheduled audience refresh
* `boundary-admin` → Audience builder UI

**Voorbeelden**:

* Marketing platforms: Email campaigns, push notifications
* CRM: Customer segmentation, lead scoring
* E-commerce: Promo targeting, cart abandonment campaigns
* SaaS: Feature announcements, onboarding emails
* HR platforms: Employee communications, surveys

**Success Criteria**:

* [ ] Audience definitie via `defaudience` macro
* [ ] Built-in filter types (10-15 types: demographics, location, behavior, tenure)
* [ ] Dynamic audience building (runtime)
* [ ] Caching layer (precompute, TTL, refresh)
* [ ] Query optimization (index recommendations)
* [ ] Admin UI: audience builder, preview
* [ ] Integration met push, email, forms
* [ ] AGENTS.md met filter voorbeelden

**Waarom Phase 3 na Phase 1+2?**

2. **Complementary to B2B**: Phase 1+2 focuste op B2B, Phase 3 focust op B2C/mobile

[cols="1,3"]

* [ ] Admin UI: form builder (drag-and-drop)
* [ ] AGENTS.md met conditional logic voorbeelden

* Average savings per project: €30k

**Calculation**:

* Savings: 35 × €30k = **€1.050.000**
* Investment: €102.4k
* **Net ROI**: €947.600 (925%)

**Conservative Scenario** (50% adoption, 40 projecten):

* Savings: 20 × €30k = **€600.000**
* Investment: €102.4k
* **Net ROI**: €497.600 (485%)

=== Per-Phase Risks

==== Phase 1 Risks

* **Workflow complexity**: State machines zijn moeilijk - invest in tests
* **Search performance**: PostgreSQL FTS moet voldoende zijn - benchmark early

**Mitigation**: Thorough design phase, POC voor critical components

==== Phase 2 Risks

* **PDF/Excel quality**: Formatting issues - use battle-tested Java libs (Apache PDFBox, POI)
* **Calendar timezone bugs**: DST handling is notorisch moeilijk - comprehensive test suite

**Mitigation**: Use existing Java date/time libraries (java.time), extensive timezone tests

==== Phase 3 Risks

* **Firebase/APNs integration**: Multi-platform push is complex - invest in integration tests
* **Audience query performance**: Complex filters kunnen traag zijn - optimize early

**Mitigation**: Thorough integration testing, performance benchmarks, caching layer

==== Phase 4 Risks

* **Low usage**: Libraries worden niet gebruikt - delay Phase 4 tot concrete behoefte
* **Form builder UI complexity**: Drag-and-drop is complex frontend work

**Mitigation**: GO/NO-GO decision na Phase 3 based on actual usage data

== Decision Points

=== Go/No-Go Gates

==== After Phase 1 (Q2 2026)

**Evaluate**:

* Did 3+ projects adopt boundary-workflow or boundary-search?
* Was performance acceptable (<50ms search, workflow transitions smooth)?
* Was maintenance burden acceptable (<10 dagen/quarter)?

**If NO** → Pause, reassess approach +
**If YES** → Proceed to Phase 2

==== After Phase 2 (Q3 2026)

**Evaluate**:

* Did 2+ projects adopt boundary-reports or boundary-calendar?
* Were enterprise features compelling in sales?
* Was cumulative ROI on track (€40k+ savings)?

**If NO** → Pause, reassess priorities +
**If YES** → Proceed to Phase 3

==== After Phase 3 (Q4 2026)

**Evaluate** (CRITICAL for Phase 4 decision):

* How many 2026 projects had mobile apps with push notifications?
* Did projects use complex audience targeting?
* Were there geo features or dynamic forms needs?
* Was cumulative ROI on track (€70k+ savings)?

**Phase 4 GO Criteria**:

* [ ] ≥2 projecten in 2026 hadden geo features nodig
* [ ] ≥1 projecten in 2026 hadden dynamic forms nodig
* [ ] Compliance redenen (data privacy) vereisen self-hosted forms
* [ ] Team heeft capacity (geen andere priorities)

**If GO** → Implement Phase 4 in Q1 2027 +
**If NO-GO** → Defer Phase 4, revisit in Q2 2027

== Aanbevelingen

=== Primary Recommendation: GO voor Phases 1-3

**Actie**: Implementeer 6 high-priority libraries in 2026 (Q2-Q4).

**Investment**: €79.200 (99 development days)

**Rationale**:

* ✅ Hoogste ROI libraries (€4k-€12k per project)
* ✅ Adresseren 80% B2B + 60% B2C needs
* ✅ Fast breakeven (2.6 projecten = 4-6 maanden)
* ✅ Complete B2B + B2C/mobile story

**Expected Return**: Na 4 projecten → €100k-€160k savings

=== Secondary Recommendation: Defer Phase 4

**Actie**: Evalueer Phase 4 na Q4 2026 based on actual usage data.

**Rationale**:

* ⚠️ Laagste ROI (€400-€4k per project)
* ⚠️ Niche use cases (20-40% adoption)
* ⚠️ Long-term breakeven (10+ projecten)
* ⚠️ Third-party alternatieven acceptabel (Google Maps, Typeform)

**Decision Criteria**:

* If ≥2 projecten in 2026 had geo features nodig → GO
* If ≥1 projecten had dynamic forms nodig + compliance redenen → GO
* Otherwise → NO-GO (revisit Q2 2027)

=== Tertiary Recommendation: Monitor & Iterate

**Actie**: Track library usage + ROI quarterly.

**Metrics**:

* Libraries gebruikt per project (% adoption)
* Development time savings (geschat vs werkelijk)
* Bugs/issues per library (maintenance burden)
* Feature requests (extension needs)

**Review Quarterly**:

* Adjust priorities based on actual usage
* Deprecate unused libraries (if adoption <20%)
* Expand high-value libraries (if adoption >80%)

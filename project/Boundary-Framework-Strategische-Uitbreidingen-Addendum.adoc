=== Context: OTTO Workforce Backend Analyse

De analyse van de OTTO Workforce Backend (myOTTO platform) heeft belangrijke inzichten opgeleverd over functionaliteit die **complementair is aan de WBA analyse**. Waar WBA B2B enterprise patterns onthulde (workflows, search, reports), onthult OTTO **B2C/employee-facing patterns**:

.OTTO Applicatie Kenmerken
* **Domein**: Workforce management platform voor flexwerkers
* **Codebase**: ~30k lines Python, Vue 2 frontend, mobile apps (iOS/Android)
* **Stack**: Django 5.2, Firebase Cloud Messaging, django-q2, Redis, MariaDB
* **Users**: Tienduizenden flexwerkers (B2C context)
* **Key Features**: Push notifications, surveys, complex user targeting, scheduling, multi-language

**Verschil met WBA**:

* WBA = B2B (event management voor culturele werkbezoeken)
* OTTO = B2C (employee self-service platform)

**Complementary Gaps**: OTTO onthult communication/engagement patterns die WBA niet had.

=== Probleem Beschrijving

==== Wat OTTO Heeft (Custom Firebase Implementation)

OTTO gebruikt **Firebase Cloud Messaging (FCM)** voor push notifications naar iOS/Android apps:

[source,python]
----
# Django model voor push notifications
class PushNotification(models.Model):
    WAITING = "W"
    IN_PROGRESS = "I"
    FAILED = "F"
    SENT = "D"
    
    STATUSES = (
        (WAITING, "Waiting to send push"),
        (IN_PROGRESS, "Sending push in progress"),
        (FAILED, "Failed to send push"),
        (SENT, "Sent push"),
    )
    
    payload = models.TextField()
    details = models.ForeignKey(PushNotificationDetails, on_delete=models.CASCADE)
    response = models.TextField()
    number_of_receivers = models.IntegerField()
    status = models.CharField(choices=STATUSES, default=WAITING, max_length=1)

# Sending pushes
def send_fcm_push(registration_id, message, collapse_key, add_data=None):
    """Send a message to a single registration ID."""
    result = FcmPush(message, [registration_id], collapse_key, add_data=add_data).send()

# Scheduled pushes
def async_send_push(otto_user, collapse_key, message, push_at=None, 
                    origin_url_name=None, extra_data=None):
    """Schedule or send push immediately."""
    if push_at and push_at > timezone.now():
        async_task("notifications.tasks.send_fcm_push",
                   schedule_type="O",
                   schedule_at=push_at,
                   registration_id=otto_user.registration_id,
                   message=message,
                   collapse_key=collapse_key)
    else:
        async_task("notifications.tasks.send_fcm_push", ...)
----

**Complexiteit in OTTO**:

* ~500 lines push notification logic
* Firebase SDK integratie (firebase-admin Python package)
* Multi-language support (translate message per user's preferred language)
* Deep linking (origin_url in payload voor navigate naar specifieke screen)
* Collapse keys (groeperen van notifications)
* Silent notifications (data-only, no alert)
* Scheduled pushes (via django-q2)
* Batch sending (max 500 tokens per Firebase request)
* Error handling & retry logic
* Status tracking (waiting, sending, sent, failed)

==== Waarom Dit Een Strategische Gap Is

[cols="1,3"]
|===
|Aspect |Rationale

|**Ubiquity**
|80% van consumer-facing apps heeft push notifications (mobiele apps, PWAs)

|**Complexity**
|Push notifications zijn *multi-platform* (FCM voor Android, APNs voor iOS), require device registration, token management, error handling

|**Engagement Impact**
|Push notifications increase retention by 3-10x vs email-only (Leanplum study 2019)

|**Boilerplate**
|Zonder library schrijf je per project 500-1000 lines boilerplate voor push logic

|**Vendor Lock-in**
|Zonder eigen library grijpen teams naar externe services (OneSignal, Pusher, Airship) met maandelijkse kosten
|===

**Critical for Mobile Apps**: Boundary wil competitive zijn voor **mobile backend APIs** - push is essentieel.

=== Voorgestelde Oplossing: boundary-push

==== Library Design

**Core Abstractions**:

[source,clojure]
----
(ns boundary.push.core)

;; Multi-provider support (Firebase, APNs, OneSignal, etc.)
{:boundary/push-service
 {:providers {:fcm {:enabled? true
                    :credentials-path "firebase-credentials.json"
                    :batch-size 500}  ; Firebase limit
              :apns {:enabled? true
                     :key-path "apns-key.p8"
                     :key-id "ABC123"
                     :team-id "DEF456"
                     :topic "com.myapp.push"}}
  :default-provider :fcm
  :jobs-service :boundary/jobs-service  ; Voor scheduled pushes
  :cache-service :boundary/cache-service}}  ; Token caching

;; Declaratieve push definitie
(defpush welcome-notification
  {:title "Welcome to MyApp!"
   :body "Start exploring features..."
   :providers [:fcm :apns]  ; Multi-platform
   :deep-link {:route :dashboard :params {}}  ; Deep linking
   :data {:user-id "{{user_id}}"
          :notification-type "welcome"}
   :collapse-key "welcome"  ; Group similar notifications
   :priority :high
   :ttl 86400})  ; 24 hours time-to-live

;; Usage: Send to single user
(push/send! welcome-notification
            {:user-id user-id
             :device-token "fcm-token-abc123"
             :platform :android})

;; Send to multiple users (batch)
(push/send-batch! welcome-notification
                  {:user-ids [user1 user2 user3]
                   :language :nl})  ; Auto-translate title/body

;; Schedule push for future
(push/schedule! welcome-notification
                {:user-id user-id
                 :send-at (instant/plus (instant/now) (duration/days 1))})

;; Send to topic (broadcast)
(push/send-to-topic! "news-updates"
                     {:title "Breaking News"
                      :body "..."
                      :platform :all})
----

=== Wanneer NIET boundary-push gebruiken?

De library is bedoeld voor **self-hosted push notifications**. Gebruik third-party services direct als:

* ❌ Project heeft <1000 users (OneSignal free tier voldoende)
* ❌ Compliance vereist managed service (SOC2/HIPAA certified third-party)
* ❌ Advanced features nodig: A/B testing, journey builder, in-app messaging (third-party services bieden dit)

Voor deze gevallen kan de library wél een adapter bieden (zelfde API, andere backend zoals OneSignal).

==== Wat OTTO Heeft (Complex User Targeting)

OTTO heeft **complexe audience targeting** voor notifications en surveys:

[source,python]
----
# Django model voor notification targeting
class Notification(models.Model):
    # Notification filter settings
    paris_ids = models.TextField(blank=True)  # Employee IDs
    exclude_invalid_contracts = models.BooleanField(default=False)
    
    clients = models.ManyToManyField(Account, blank=True)
    clients_start_date = models.DateField(blank=True, null=True)
    clients_end_date = models.DateField(blank=True, null=True)
    
    work_locations = models.ManyToManyField(WorkLocation, blank=True)
    work_locations_start_date = models.DateField(blank=True, null=True)
    work_locations_end_date = models.DateField(blank=True, null=True)
    
    housing_locations = models.ManyToManyField(HousingLocation, blank=True)
    housing_locations_start_date = models.DateField(blank=True, null=True)
    housing_locations_end_date = models.DateField(blank=True, null=True)
    
    min_age = models.IntegerField(blank=True, null=True)
    max_age = models.IntegerField(blank=True, null=True)
    bsn_present = models.BooleanField(null=True)  # Tax ID present?
    week_number = models.IntegerField(blank=True, null=True)  # Weeks employed

# Complex filtering logic
def get_receivers(self):
    """Get the receivers by applying 8 different filters."""
    receivers = OttoUser.objects.all()
    receivers = self.filter_paris_id(receivers)
    receivers = self.filter_clients(receivers)
    receivers = self.filter_work_locations(receivers)
    receivers = self.filter_housing_locations(receivers)
    receivers = self.filter_age(receivers)
    receivers = self.filter_bsn(receivers)
    receivers = self.filter_week_number(receivers)
    receivers = self.exclude_expired_contracts(receivers)
    return receivers

def filter_age(self, qs):
    """Get receivers based on age."""
    if self.min_age and self.max_age:
        current = timezone.now()
        max_datetime = current - timedelta(days=int(self.min_age * 365.25))
        min_datetime = current - timedelta(days=int((self.max_age * 365.25) + 365.25))
        return qs.filter(date_of_birth__gte=min_datetime, date_of_birth__lte=max_datetime)
    return qs

# ~300 lines van filter logic
----

**Complexiteit**:

* 8 verschillende filter types (IDs, locations, age, tenure, contracts)
* Date range filters (work_locations tussen start/end datum)
* Complex date queries (age calculation, weeks employed)
* Filter combinations (AND/OR logic)
* **Pattern herhaalt zich**: Notification model heeft dezelfde filters als Survey model (code duplication)

==== Library Design

**Core Abstractions**:

(ns boundary.audience.core)

;; Declaratieve audience builder
(defaudience active-employees
  {:name "Active Employees (30 days)"
   :description "Users with contract active in last 30 days"
   
   :filters [{:type :contract-status
              :operator :active
              :date-range {:start (instant/minus (instant/now) (duration/days 30))
                           :end (instant/now)}}]})

(defaudience young-amsterdam-employees
  {:name "Young employees in Amsterdam"
   :description "Age 18-30, worked in Amsterdam in last 3 months"
   
   :filters [{:type :age-range
              :min 18
              :max 30}
             
             {:type :work-location
              :locations [:amsterdam :amstelveen]
              :date-range {:start (instant/minus (instant/now) (duration/days 90))
                           :end (instant/now)}}]
   
   :combinator :and})  ; All filters must match (default)

;; Complexe audience met OR logic
(defaudience high-value-users
  {:name "High Value Users"
   
   :filters [{:type :or
              :children [{:type :order-count
                          :operator :gte
                          :value 10}
                         
                         {:type :lifetime-value
                          :operator :gte
                          :value 1000}
                         
                         {:type :subscription-tier
                          :operator :eq
                          :value :premium}]}]})

;; Usage: Evaluate audience
(audience/resolve active-employees)
;; => {:user-ids [uuid1 uuid2 uuid3 ...]
;;     :count 150
;;     :cached? false}

;; Usage: Count zonder ophalen (performance)
(audience/count active-employees)
;; => 150

;; Usage: Preview (debug)
(audience/explain active-employees)

==== Database Schema

[source,sql]
----
-- Form definitions
CREATE TABLE forms (
  id UUID PRIMARY KEY,
  key TEXT NOT NULL UNIQUE,  -- From defform
  title TEXT NOT NULL,
  description TEXT,
  pages JSONB NOT NULL,  -- Array of pages with questions
  settings JSONB NOT NULL,  -- Form settings
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  archived_at TIMESTAMP
);

-- Form sends (who received which form, when)
CREATE TABLE form_sends (
  id UUID PRIMARY KEY,
  form_id UUID NOT NULL REFERENCES forms(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  sent_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  expires_at TIMESTAMP,
  mandatory BOOLEAN NOT NULL DEFAULT FALSE,
  UNIQUE(form_id, user_id)
);

-- Form responses
CREATE TABLE form_responses (
  id UUID PRIMARY KEY,
  form_send_id UUID NOT NULL REFERENCES form_sends(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  responses JSONB NOT NULL,  -- {question_id: answer}
  started_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  submitted_at TIMESTAMP,
  completion_time_seconds INTEGER,
  UNIQUE(form_send_id)
);

CREATE INDEX idx_form_responses_user ON form_responses(user_id);
CREATE INDEX idx_form_responses_submitted ON form_responses(submitted_at);

=== boundary-push: Complete Firebase Example

[source,clojure]
----
(ns myapp.notifications.push
  (:require [boundary.push.core :as push]))

;; Push definitie (from OTTO pattern)
(defpush shift-reminder
  {:title-i18n {:nl "Dienst herinnering"
                :en "Shift Reminder"
                :pl "Przypomnienie o zmianie"}
   :body-i18n {:nl "Je dienst begint over {{hours}} uur"
               :en "Your shift starts in {{hours}} hours"
               :pl "Twoja zmiana rozpoczyna się za {{hours}} godzin"}
   :providers [:fcm]
   :deep-link {:route :shift-detail :params {:shift-id "{{shift_id}}"}}
   :collapse-key "shift-reminder"
   :priority :high
   :data {:notification-type "shift-reminder"
          :shift-id "{{shift_id}}"
          :shift-start "{{shift_start}}"}})

;; Usage (OTTO pattern: scheduled 2 hours before shift)
(defn schedule-shift-reminders [shifts]
  (doseq [shift shifts]
    (let [reminder-time (instant/minus (:start-time shift) (duration/hours 2))
          employee-user-id (:employee-id shift)]
      (push/schedule! shift-reminder
                      {:user-id employee-user-id
                       :send-at reminder-time
                       :template-vars {:hours 2
                                       :shift-id (:id shift)
                                       :shift-start (str (:start-time shift))}}))))

;; Batch push (OTTO pattern: notify all employees)
(defn notify-all-employees [title body]
  (let [employee-ids (db/query "SELECT id FROM users WHERE user_type = 'employee'")]
    (push/send-batch! {:title title :body body}
                      {:user-ids employee-ids
                       :platform :all})))

=== boundary-audience: OTTO Targeting Pattern

[source,clojure]
----
(ns myapp.notifications.targeting
  (:require [boundary.audience.core :as audience]))

;; Replicate OTTO's complex targeting
(defaudience amsterdam-new-employees
  {:name "Amsterdam New Employees (Week 6)"
   :description "Employees in Amsterdam in their 6th week"
   
   :filters [;; Week number filter (OTTO: week_number = 6)
             {:type :tenure
              :operator :eq
              :value {:weeks 6}
              :tolerance {:weeks 1}}  ; Week 6 ± 1 week
             
             ;; Work location filter (OTTO: work_locations + date range)
             {:type :work-location
              :locations [:amsterdam :amstelveen]
              :date-range {:start (instant/minus (instant/now) (duration/days 90))
                           :end (instant/now)}}
             
             ;; Contract validity filter (OTTO: exclude_invalid_contracts)
             {:type :contract-status
              :operator :active
              :date-range {:start (instant/now)
                           :end (instant/plus (instant/now) (duration/days 30))}}
             
             ;; Age range (OTTO: min_age, max_age)
             {:type :age-range
              :min 18
              :max 30}]
   
   :combinator :and})

;; Usage: Send targeted survey (OTTO pattern)
(defn send-employee-survey [survey-id]
  (let [audience (audience/resolve amsterdam-new-employees)
        user-ids (:user-ids audience)]
    (forms/send-survey! survey-id {:user-ids user-ids
                                    :mandatory? true
                                    :expires-at (instant/plus (instant/now) (duration/days 7))})))
----

(ns myapp.surveys.definitions

                :en "Employee Satisfaction - Week 6"
                :pl "Zadowolenie pracowników - Tydzień 6"}

                      :en "Help us improve your experience"
                      :pl "Pomóż nam poprawić Twoje doświadczenie"}
   
   :pages [{:id :experience
            :title-i18n {:nl "Je ervaring" :en "Your experience" :pl "Twoje doświadczenie"}

                         :type :rating
                         :label-i18n {:nl "Hoe tevreden ben je overall?"
                                      :en "How satisfied are you overall?"
                                      :pl "Jak jesteś ogólnie zadowolony?"}

                                               5 {:nl "Zeer tevreden" :en "Very satisfied" :pl "Bardzo zadowolony"}}}

                        {:id :would-recommend
                         :type :yes-no

                                      :en "Would you recommend OTTO?"
                                      :pl "Czy poleciłbyś OTTO?"}
                         :required? true}

                         :type :long-text

                                      :en "What could we improve?"
                                      :pl "Co możemy poprawić?"}

                                            :pl "Podziel się swoimi przemyśleniami..."}

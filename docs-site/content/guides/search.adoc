= Full-Text Search

*Index any domain entity and search it with a single Clojure call.*

The `boundary-search` module provides document indexing, full-text search, and autocomplete suggestions.
It uses PostgreSQL native FTS (`to_tsvector` / `ts_rank`) in production and automatically falls back to
a `LIKE`-based query on H2 and SQLite (development / CI).
No external search service is required.

== Quick Example

[source,clojure]
----
(require '[boundary.search.core.index :as search]
         '[boundary.search.ports :as search-ports])

;; 1. Define an index (once, at module level)
(search/defsearch product-search
  {:id          :product-search
   :entity-type :product
   :language    :english
   :fields      [{:name :title       :weight :A}
                 {:name :description :weight :B}
                 {:name :tags        :weight :C}]
   :filters     [:tenant-id :status]})  ; optional: declare filterable keys

;; 2. Index a document when you save an entity
(search-ports/index-document!
  search-svc
  :product-search
  product-uuid
  {:title "Widget Pro" :description "A professional widget" :tags ["tools"]}
  {:metadata     {:sku "WGT-001"}
   :filter-values {:tenant-id "acme" :status "active"}})

;; 3. Search (with optional filter to scope results)
(search-ports/search search-svc :product-search "widget"
                     {:limit 20 :offset 0 :highlight? true
                      :filters {:tenant-id "acme"}})
;; => {:results [{:entity-type :product
;;               :entity-id   #uuid "..."
;;               :rank        0.92
;;               :snippet     "A professional <b>widget</b>..."}]
;;     :total 1 :query "widget" :took-ms 4}
----

== Core Concepts

=== Defining a Search Index

`defsearch` is a macro that binds a var *and* registers the definition in a global in-process registry:

[source,clojure]
----
(search/defsearch order-search
  {:id          :order-search
   :entity-type :order
   :language    :english
   :fields      [{:name :reference :weight :A}
                 {:name :customer  :weight :B}
                 {:name :notes     :weight :C}]
   :filters     [:tenant-id :status]})  ; declare filterable dimensions
----

`:filters` declares which keyword keys will be stored as filter dimensions.
Clients pass their values at index time via `:filter-values` and query time via `:filters`.

Field weights control PostgreSQL FTS ranking (`:A` highest, `:D` lowest):

[cols="1,1,3"]
|===
|Weight |Priority |Typical use

|`:A` |Highest |Name, title, reference
|`:B` |High |Summary, customer name
|`:C` |Normal |Body text, tags
|`:D` |Lowest |Auxiliary notes
|===

=== Indexing Documents

Call `index-document!` from your service layer whenever an entity is created or updated:

[source,clojure]
----
;; In your module's service
(defn create-order! [search-svc order]
  (let [saved (persist-order! order)]
    (search-ports/index-document!
      search-svc
      :order-search
      (:id saved)
      {:reference (:reference saved)
       :customer  (full-name (:customer saved))
       :notes     (:notes saved)}
      ;; optional opts
      {:metadata      {:order-ref (:reference saved)}
       :filter-values {:tenant-id (str (:tenant-id saved))
                       :status    (name (:status saved))}})
    saved))

;; Remove the document when the entity is deleted
(search-ports/remove-document! search-svc :order-search order-uuid)
----

`:filter-values` accepts a map of keyword → string-coercible value.
Keys are stored as snake_case JSON (`tenant-id` → `tenant_id`).

=== Searching

[source,clojure]
----
;; Full-text search with pagination
(search-ports/search search-svc :order-search "widget"
                     {:limit 20 :offset 0 :highlight? true})

;; Scoped search — only documents that match all filter values are returned
(search-ports/search search-svc :order-search "widget"
                     {:limit 20 :filters {:tenant-id "acme" :status "active"}})

;; Autocomplete / typeahead suggestions
(search-ports/suggest search-svc :order-search "wid" {:limit 5})
;; => [{:entity-type :order :entity-id #uuid "..." :rank 0.8}]
----

NOTE: When `:filters` is provided, documents that do not have *all* the
specified filter keys are excluded — there is no partial match.

The `SearchResponse` shape returned by `search`:

[source,clojure]
----
{:results [{:entity-type :product    ; keyword
            :entity-id   #uuid "..."  ; UUID
            :rank        0.92         ; float, PostgreSQL ts_rank
            :snippet     "..."        ; highlighted text (when :highlight? true)
            :metadata    {...}}]      ; optional, stored at index time
 :total   42                          ; total matching documents
 :query   "widget"                    ; sanitised query string
 :took-ms 5}                          ; query duration in milliseconds
----

== HTTP API

The module mounts two sets of routes automatically.

=== REST API (`/api/v1`)

[cols="1,2,2"]
|===
|Method |Path |Description

|`POST` |`/search/:index-id` |Full-text search
|`POST` |`/search/:index-id/suggest` |Autocomplete suggestions
|`POST` |`/search/documents` |Index a document
|`DELETE` |`/search/documents/:entity-type/:entity-id` |Remove a document
|===

Example search request:

[source,bash]
----
curl -X POST http://localhost:3000/api/v1/search/product-search \
  -H 'Content-Type: application/json' \
  -d '{"query": "widget", "limit": 20, "offset": 0, "highlight?": true}'
----

With filter scoping:

[source,bash]
----
curl -X POST http://localhost:3000/api/v1/search/product-search \
  -H 'Content-Type: application/json' \
  -d '{"query": "widget", "limit": 20, "filters": {"tenantId": "acme", "status": "active"}}'
----

Indexing a document with filter values:

[source,bash]
----
curl -X POST http://localhost:3000/api/v1/search/documents \
  -H 'Content-Type: application/json' \
  -d '{"indexId": "product-search", "entityId": "uuid-here",
       "fields": {"title": "Widget Pro"},
       "filterValues": {"tenantId": "acme", "status": "active"}}'
----

=== Admin Web UI (`/web/admin`)

[cols="1,2,2"]
|===
|Method |Path |Description

|`GET`  |`/search`                        |List all registered indices
|`GET`  |`/search/:index-id`              |Index detail + live search test form
|`POST` |`/search/:index-id/search`       |HTMX search-results fragment
|===

== Configuration

Enable in your environment config:

[source,edn]
----
;; resources/conf/dev/config.edn
{:boundary/search {:enabled? true}}
----

No additional options are needed. The module auto-detects the database dialect from
`:boundary/db-context` and selects PostgreSQL FTS or the LIKE fallback accordingly.

== Database Migration

The module creates a single `search_documents` table:

[source,sql]
----
CREATE TABLE search_documents (
  id          TEXT NOT NULL PRIMARY KEY,
  index_id    TEXT NOT NULL,
  entity_type TEXT NOT NULL,
  entity_id   TEXT NOT NULL,
  language    TEXT NOT NULL DEFAULT 'english',
  weight_a    TEXT NOT NULL DEFAULT '',    -- highest priority field(s)
  weight_b    TEXT NOT NULL DEFAULT '',
  weight_c    TEXT NOT NULL DEFAULT '',
  weight_d    TEXT NOT NULL DEFAULT '',    -- lowest priority
  content_all TEXT NOT NULL DEFAULT '',    -- all fields joined (LIKE fallback)
  metadata    TEXT,                        -- pr-str EDN map
  filters     TEXT,                        -- JSON filter dimensions
  updated_at  TEXT NOT NULL,
  UNIQUE (index_id, entity_id)
);
----

Re-indexing an entity is always safe: the upsert uses
`ON CONFLICT (index_id, entity_id) DO UPDATE SET ...`, which also updates `filters`.

The `filters` column stores a compact JSON string: `{"tenant_id":"acme","status":"active"}`.
Keys use snake_case derived from the Clojure keyword (`:tenant-id` → `"tenant_id"`).

Run migrations with:

[source,bash]
----
clojure -M:migrate migrate
----

If you are adding filters to an existing installation, apply the schema migration:

[source,bash]
----
# Adds the filters column
clojure -M:migrate migrate   # picks up 20260312000000-search-filters.up.sql
----

== Architecture

The module follows the Functional Core / Imperative Shell pattern:

[source,text]
----
libs/search/src/boundary/search/
├── schema.clj          # Malli schemas (SearchDocument, SearchResult, SearchResponse)
├── ports.clj           # ISearchStore + ISearchEngine protocols
├── core/
│   ├── index.clj       # defsearch macro, global registry, build-document
│   ├── query.clj       # SQL builders: PostgreSQL FTS + LIKE fallback
│   └── ui.clj          # Pure Hiccup components (admin UI)
└── shell/
    ├── persistence.clj  # SearchStore (next.jdbc + HoneySQL)
    ├── service.clj      # SearchService orchestration
    ├── http.clj         # Ring handlers + route definitions
    └── module_wiring.clj  # Integrant :boundary/search key
----

=== Dual Query Strategy

[cols="1,3"]
|===
|Database |Strategy

|PostgreSQL |`to_tsvector` / `plainto_tsquery` / `ts_rank` / `ts_headline`
|H2, SQLite  |`LOWER(content_all) LIKE LOWER('%q%')`
|===

The adapter is selected at startup from the datasource dialect.

= AI Tooling

*Framework-aware AI assistance: NL scaffolding, error explanation, test generation, SQL copilot, and documentation wizard.*

The `boundary-ai` library brings AI into your development workflow without leaving your terminal — and without sending your code to a cloud service by default. It runs offline-first through Ollama (a local model server), with an optional fallback to Anthropic or OpenAI when Ollama is unavailable.

What makes it *framework-aware* is that every prompt is pre-loaded with Boundary's own rules: FC/IS layer separation, kebab/snake/camelCase naming conventions, Malli schema patterns, HoneySQL syntax, and Integrant lifecycle. The AI knows the codebase conventions before you type a single word.

== Quick Example

[source,bash]
----
# Natural language scaffolding
bb scaffold ai "product module with name string required, price decimal required, stock integer"

# Skip confirmation and generate immediately
bb scaffold ai "product module with name string required, price decimal required, stock integer" --yes

# Explain the last exception in your REPL
bb ai explain < stacktrace.txt

# Generate tests for a source file
bb ai gen-tests libs/user/src/boundary/user/core/validation.clj

# HoneySQL query from English
bb ai sql "find all active users who placed an order in the last 7 days"

# Generate AGENTS.md for a library
bb ai docs --module libs/user --type agents
----

''''

== Prerequisites

=== Ollama (offline, no API key)

[source,bash]
----
# macOS / Linux
brew install ollama       # or: curl -fsSL https://ollama.ai/install.sh | sh
ollama serve              # starts at http://localhost:11434

# Pull the recommended coding model
ollama pull qwen2.5-coder:7b
----

=== Cloud providers (optional fallback)

[source,bash]
----
# Anthropic (Claude)
export ANTHROPIC_API_KEY="sk-ant-..."

# OpenAI (GPT)
export OPENAI_API_KEY="sk-..."
----

The CLI auto-detects which provider to use based on environment variables.
Ollama is always tried first if it is reachable. See <<configuration>> for Integrant wiring.

== Feature 1: Natural Language Scaffolding

Generate a complete, production-ready module from a plain English description.

[source,bash]
----
bb scaffold ai "product module with name string required, sku string required unique, \
                price decimal required, description text, stock integer"
----

The AI returns a module spec, shows you a preview (identical to the interactive wizard summary), and asks for confirmation before writing any files.
If you want non-interactive mode (for scripts/CI), pass `--yes` to skip the confirmation prompt.

[source]
----
┌─────────────────────────────────────────────────────────┐
│  Module: product                                         │
│  Entity: Product                                         │
│  Fields: name:string:required                            │
│          sku:string:required:unique                      │
│          price:decimal:required                          │
│          description:text                                │
│          stock:integer                                   │
│  Interfaces: REST API + Web UI                           │
└─────────────────────────────────────────────────────────┘

Generate this module? [y/N]
----

Internally this delegates to the same `generate-module` pipeline as the interactive `bb scaffold` wizard, so the generated files are identical in quality.

=== Field type hints

The AI understands natural language field descriptions and maps them to Boundary field types:

[cols="2,1",options="header"]
|===
| Natural language phrase | Field type
| "name", "title", "label" | `string`
| "description", "body", "content", "notes" | `text`
| "price", "amount", "cost", "salary" | `decimal`
| "count", "quantity", "stock", "score" | `integer`
| "active", "enabled", "published", "visible" | `boolean`
| "created at", "updated at", "scheduled for" | `timestamp`
| "user", "owner", "author" + "id" | `uuid`
|===

Attributes like "required", "unique", "optional" are extracted from the description as modifiers.

== Feature 2: Error Explainer

Paste a Clojure stack trace and get a root-cause explanation with a suggested fix — all in the context of Boundary's architecture.

=== From the terminal

[source,bash]
----
# Pipe from stdout
some-failing-command 2>&1 | bb ai explain

# From a file
bb ai explain --file /tmp/stacktrace.txt

# Inline string
bb ai explain "clojure.lang.ExceptionInfo: Schema validation failed ..."
----

=== From the REPL

[source,clojure]
----
(require '[boundary.ai.shell.repl :as ai])

;; Explain the most recent exception
(ai/explain *e)

;; Explain a specific exception
(try
  (do-something!)
  (catch Exception e
    (ai/explain e)))
----

=== Example output

[source]
----
ROOT CAUSE
  Schema validation failed on :boundary.user/create-user-request.
  The field :password-hash is missing from the input map.

WHY IT HAPPENED
  The handler is passing the raw request body directly to the service.
  Boundary's case-conversion middleware converts camelCase JSON keys to
  kebab-case, but this route is missing the :camel->kebab interceptor.

SUGGESTED FIX (shell/http.clj, line 42)
  Add the case-conversion interceptor to the route definition:
    {:interceptors [interceptors/camel->kebab-request]}

FC/IS NOTE
  The validation belongs in core/validation.clj (pure), which it is.
  The interceptor belongs in the shell layer — do not call it from core/.
----

== Feature 3: Test Generator

Generate a complete test namespace for any source file. The AI infers the correct test type (`^:unit` for `core/`, `^:integration` or `^:contract` for `shell/`) and follows Boundary's testing conventions.

[source,bash]
----
# Generate tests for a pure core file (produces ^:unit tests)
bb ai gen-tests libs/user/src/boundary/user/core/validation.clj

# Generate tests for a shell service (produces ^:integration tests)
bb ai gen-tests libs/user/src/boundary/user/shell/service.clj
----

The output is printed to stdout. Review it, then write it to the test directory:

[source,bash]
----
bb ai gen-tests libs/user/src/boundary/user/core/validation.clj \
  > libs/user/test/boundary/user/core/validation_test.clj
----

=== What the generator produces

For a `core/` source file:

[source,clojure]
----
(ns boundary.user.core.validation-test
  (:require [clojure.test :refer [deftest is testing]]
            [boundary.user.core.validation :as v]))

(deftest validate-email-test
  ^:unit
  (testing "accepts valid email addresses"
    (is (nil? (v/validate-email "user@example.com"))))
  (testing "rejects addresses without @"
    (is (some? (v/validate-email "not-an-email"))))
  (testing "rejects blank input"
    (is (some? (v/validate-email "")))))
----

For a `shell/` source file, the generator produces `^:integration` tests that use `reify` or `mock` adapters instead of real database connections.

== Feature 4: SQL Copilot

Describe a query in plain English and get a ready-to-use HoneySQL map plus a SQL preview.

=== Terminal

[source,bash]
----
bb ai sql "find all active users who placed an order in the last 7 days"
----

[source]
----
HONEYSQL
  {:select [:u.id :u.email :u.created-at]
   :from   [[:users :u]]
   :join   [[:orders :o] [:= :u.id :o.user-id]]
   :where  [:and
             [:= :u.active true]
             [:>= :o.created-at [:raw "NOW() - INTERVAL '7 days'"]]]
   :order-by [[:u.created-at :desc]]}

EXPLANATION
  Joins users to orders on user_id, filters for active users
  with at least one order created within the last 7 days.

RAW SQL
  SELECT u.id, u.email, u.created_at
  FROM users u
  JOIN orders o ON u.id = o.user_id
  WHERE u.active = TRUE
    AND o.created_at >= NOW() - INTERVAL '7 days'
  ORDER BY u.created_at DESC
----

=== REPL

[source,clojure]
----
(require '[boundary.ai.shell.repl :as ai])

;; Wire the service once (usually in your dev namespace)
(ai/set-service! (ig/ref :boundary/ai-service))

;; Then use it from any REPL buffer
(ai/sql "find active users with orders in the last 7 days")
;; => {:honeysql {...} :explanation "..." :raw-sql "SELECT ..."}
----

The copilot reads all `schema.clj` files in your project to understand your actual table and field names, so the generated queries use your real column names instead of generic placeholders.

== Feature 5: Documentation Wizard

Generate three kinds of documentation from your source files:

[cols="1,2",options="header"]
|===
| Type | Output
| `agents` | `AGENTS.md` — Purpose, Key Namespaces table, Common Pitfalls, Testing commands
| `openapi` | OpenAPI 3.x YAML — inferred from Reitit routes and Malli schemas
| `readme` | `README.md` — Description, usage, configuration, dependencies
|===

[source,bash]
----
# AGENTS.md for a library
bb ai docs --module libs/user --type agents

# OpenAPI spec
bb ai docs --module libs/user --type openapi > docs/openapi/user.yaml

# README
bb ai docs --module libs/user --type readme > libs/user/README.md

# All three at once
bb ai docs --module libs/user
----

`--module` accepts any path: `libs/user`, `src/boundary/invoices`, etc.

== Configuration

=== Integrant (dev config)

[source,clojure]
----
;; resources/conf/dev/config.edn
{:boundary/ai-service
 {:provider :ollama
  :model    #or [#env AI_MODEL "qwen2.5-coder:7b"]
  :base-url #or [#env OLLAMA_URL "http://localhost:11434"]
  :fallback {:provider :anthropic
             :model    #or [#env AI_MODEL "claude-haiku-4-5-20251001"]
             :api-key  #env ANTHROPIC_API_KEY}}}
----

=== Integrant (prod config)

[source,clojure]
----
;; resources/conf/prod/config.edn
{:boundary/ai-service
 {:provider :anthropic
  :model    #or [#env AI_MODEL "claude-sonnet-4-6"]
  :api-key  #env ANTHROPIC_API_KEY}}
----

=== Tests (no-op provider)

[source,clojure]
----
;; resources/conf/test/config.edn
{:boundary/ai-service
 {:provider :no-op}}
----

The no-op provider returns deterministic canned responses so your tests never make network calls and never need an API key.

=== Provider selection in CLI tools

When you run `bb ai ...` outside of an Integrant system, the CLI builds a provider from environment variables:

[cols="1,2",options="header"]
|===
| Priority | Condition
| 1st | `ANTHROPIC_API_KEY` is set → use Anthropic
| 2nd | `OPENAI_API_KEY` is set → use OpenAI
| 3rd (default) | Use Ollama at `$OLLAMA_URL` (default: `http://localhost:11434`)
|===

== Using in application code

The service functions are available directly from `boundary.ai.shell.service`:

[source,clojure]
----
(ns myapp.ai-handler
  (:require [boundary.ai.shell.service :as ai]))

;; ai-service comes from your Integrant system map
(defn handle-explain-request [ai-service request]
  (let [stacktrace (get-in request [:body :stacktrace])
        result     (ai/explain-error ai-service stacktrace ".")]
    (if (:error result)
      {:status 500 :body {:error (:error result)}}
      {:status 200 :body {:explanation (:text result)}})))
----

All service functions accept an optional trailing `opts` map that is forwarded to the provider (e.g. `{:model "..."}` to override the configured model for a single call).

== Testing

[source,bash]
----
# All AI tests (unit + integration, no live network calls)
clojure -M:test:db/h2 :ai

# Unit tests only (pure core functions)
clojure -M:test:db/h2 --focus-meta :unit :ai

# Integration tests (mock IAIProvider)
clojure -M:test:db/h2 --focus-meta :integration :ai

# Contract tests against live Ollama (requires Ollama running)
OLLAMA_URL=http://localhost:11434 clojure -M:test:db/h2 --focus-meta :contract :ai

# Lint
clojure -M:clj-kondo --lint libs/ai/src libs/ai/test
----

=== Writing tests with a mock provider

[source,clojure]
----
(ns myapp.ai-test
  (:require [clojure.test :refer [deftest is testing]]
            [boundary.ai.ports :as ports]
            [boundary.ai.shell.service :as ai]))

(def mock-provider
  (reify ports/IAIProvider
    (complete [_ _messages _opts]
      {:text "mock explanation" :tokens 10 :provider :mock :model "mock"})
    (complete-json [_ _messages _schema _opts]
      {:data {:module-name "product" :entity "Product" :fields []}
       :tokens 20 :provider :mock :model "mock"})
    (provider-name [_] :mock)))

(def test-service {:provider mock-provider :fallback nil})

(deftest explain-error-test
  ^:integration
  (testing "returns explanation text"
    (let [result (ai/explain-error test-service "some stacktrace" ".")]
      (is (string? (:text result)))
      (is (not (:error result))))))
----

== Common pitfalls

[arabic]
. *Ollama not running* — `bb ai ...` will print `primary AI provider failed` and retry with the fallback if configured. Start Ollama with `ollama serve` before running commands offline.

. *Model not pulled* — `ollama pull qwen2.5-coder:7b` must be run once before first use. Using an unpulled model name causes a 404 from Ollama's API (surfaced as `{:error "..."}`).

. *Anthropic system messages* — The Anthropic API does not accept `:system` role inside the messages array. The `AnthropicProvider` adapter handles this automatically by extracting system messages and passing them as the top-level `:system` field. You do not need to handle this yourself when calling `boundary.ai.shell.service` functions.

. *`complete-json` is best-effort* — Structured JSON output relies on the model following a prompt instruction. Ollama and Anthropic do not guarantee strict schema adherence. Always validate the returned `:data` map with Malli before using it in application logic.

. *Test provider isolation* — Use `:no-op` in `test/config.edn` (already configured). If you write tests that require specific AI behaviour, use `reify ports/IAIProvider` directly as shown above. Never use the real Ollama or cloud providers in `^:unit` or `^:integration` tests.

. *FC/IS boundary* — `boundary.ai.shell.service` is in the shell layer. Do not call it from `core/` namespaces. The pure prompt builders in `boundary.ai.core.prompts` and the context extractors in `boundary.ai.core.context` are safe to call from pure functions.

. *Overlarge source files* — The context extractor (`ctx/truncate-source`) truncates source files at 150 lines before sending them to the AI. If your file is larger, the AI works on the truncated version. Split large files into focused namespaces for best results.

. *Rate limits on cloud providers* — Anthropic and OpenAI enforce per-minute token limits. If you run `bb ai docs --module libs/platform` (a large module), you may hit a rate limit. In that case, run `--type agents`, `--type openapi`, and `--type readme` as separate calls spaced a few seconds apart.

= Building a Task Management API

Welcome to the comprehensive hands-on tutorial for the Boundary Framework. In this tutorial you will build a production-ready Task Management API from scratch.

*Time:* 1–2 hours · *Level:* Beginner to Intermediate

'''

== What You'll Build

A "TaskMaster" API that:

. Registers and authenticates users (JWT-based)
. Creates, reads, updates, and deletes (CRUD) tasks
. Validates all inputs using Malli schemas
. Implements business rules in the Functional Core
. Passes a complete test suite

'''

== Chapter 1: Introduction & Prerequisites

=== The Boundary Philosophy

Boundary is designed around the *Functional Core / Imperative Shell* (FC/IS) paradigm:

* *Functional Core* — All business logic in pure, side-effect-free functions. Easy to test and reason about.
* *Imperative Shell* — All I/O (database, HTTP, logging) at the edges of your system.
* *Ports* — Protocols that define contracts between layers.

=== Prerequisites

* *Java 11+* (OpenJDK or Amazon Corretto)
* *Clojure CLI*
* *Babashka* (`bb`)
* *curl* (for testing endpoints)

Verify your environment:

[source,bash]
----
java -version   # Should show 11 or higher
clojure -version
bb --version
curl --version
----

=== The Anatomy of a Module

[source,text]
----
libs/task/src/boundary/task/
├── core/
│   ├── task.clj        # Pure business logic (FC)
│   └── validation.clj  # Validation rules
├── shell/
│   ├── service.clj     # Orchestration layer (IS)
│   ├── http.clj        # HTTP handlers & routes
│   └── persistence.clj # Database implementation
├── ports.clj           # Protocol definitions
└── schema.clj          # Malli validation schemas
----

[source,bash]

=== How Interceptors Work

Boundary uses interceptors for cross-cutting concerns like authentication. In `libs/user/src/boundary/user/shell/http.clj` you'll find an auth interceptor that validates the JWT on incoming requests:

[source,clojure]
----
(def auth-interceptor
  {:name :auth
   :enter (fn [ctx]
            (let [token (get-in ctx [:request :headers "authorization"])]
              (if (valid-token? token)
                (assoc-in ctx [:request :identity] (decode-token token))
                (throw (ex-info "Unauthorized" {:type :unauthorized})))))})
----

By adding this interceptor to a route, only authenticated users can access it.

== Chapter 4: Generating the Task Module

=== Step 1: Scaffold the Module

Run the interactive wizard:

[source,bash]
----
bb scaffold
----

When prompted, enter:

* Module name: `task`
* Entity name: `Task`
* Fields: `title:string:required`, `description:string`, `priority:string:required`, `due-date:date`, `completed:boolean:required`

=== Step 2: Explore the Generated Files

==== schema.clj

Defines the data structure using *Malli*:

[source,clojure]
----
(def task-schema
  [:map
   [:id :uuid]
   [:title [:string {:min 1 :max 255}]]
   [:description {:optional true} [:maybe :string]]
   [:priority [:string {:min 1}]]
   [:due-date {:optional true} [:maybe inst?]]
   [:completed :boolean]
   [:created-at inst?]
   [:updated-at inst?]])
----

==== core/task.clj

The *Functional Core* — pure functions only:

[source,clojure]
----
(defn prepare-task [task-data]
  (let [now (java.time.Instant/now)]
    (merge task-data
           {:id (random-uuid)
            :created-at now
            :updated-at now})))
----

==== shell/service.clj

The *Imperative Shell* — orchestrates validation, core, and persistence:

[source,clojure]
----
(defn create-task! [this task-data]
  (let [validated (schema/validate-task task-data)
        prepared  (task-core/prepare-task validated)]
    (ports/save-task (:repository this) prepared)))
----

== Chapter 6: Validation & Error Handling

=== Trigger a Validation Error

Try creating a task without the required `title` field:

[source,bash]
----
curl -X POST http://localhost:3000/api/v1/tasks \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_TOKEN" \
  -d '{
    "description": "Missing title",
    "priority": "low",
    "completed": false
  }'
----

Boundary's error interceptor catches validation failures and returns a structured `400 Bad Request` response.

=== Custom Validation Rules

Open `libs/task/src/boundary/task/schema.clj` and add an enum constraint for `priority`:

[source,clojure]
----
(def task-priority-schema
  [:enum "low" "medium" "high"])

(def task-schema
  [:map
   ;; ... other fields
   [:priority task-priority-schema]])
----

After refreshing the REPL with `(ig-repl/reset)`, invalid priorities will be rejected automatically.

=== Cross-Field Validation

For validation that depends on multiple fields:

[source,clojure]
----
(def create-task-schema
  [:and
   [:map
    [:title :string]
    [:due-date {:optional true} inst?]]
   [:fn {:error/message "Due date must be in the future"}
    (fn [{:keys [due-date]}]
      (if due-date
        (.isAfter due-date (java.time.Instant/now))
        true))]])
----

== Chapter 7: Business Logic in the Functional Core

=== Define the Rule

When a task is marked completed, also record the completion date. This belongs in `core/task.clj`:

[source,clojure]
----
(defn complete-task [task]
  (-> task
      (assoc :completed true)
      (assoc :completed-at (java.time.Instant/now))
      (assoc :updated-at (java.time.Instant/now))))
----

=== Update the Shell

In `shell/service.clj`, call this core function when updating:

[source,clojure]
----
(defn update-task! [this id updates]
  (let [existing (ports/get-task (:repository this) id)]
    (if (and (:completed updates) (not (:completed existing)))
      (ports/save-task (:repository this) (task-core/complete-task existing))
      (ports/save-task (:repository this) (merge existing updates)))))
----

=== Why This Pattern?

By keeping `complete-task` in the core, it can be tested with plain Clojure maps — no database, no HTTP, no Integrant system required.

== Chapter 10: Advanced Scenarios

=== Implementing Soft Deletes

==== 1. Update the Schema

Add `deleted-at` to `schema.clj`:

[source,clojure]
----
(def task-schema
  [:map
   ;; ... existing fields
   [:deleted-at {:optional true} [:maybe inst?]]
   [:created-at inst?]
   [:updated-at inst?]])
----

==== 2. Core Logic

In `core/task.clj`:

[source,clojure]
----
(defn mark-as-deleted [task]
  (assoc task :deleted-at (java.time.Instant/now)))
----

==== 3. Persistence Layer

Update the persistence adapter to use soft-delete and filter deleted records:

[source,clojure]
----
;; DELETE becomes an UPDATE
(delete-task [this id]
  (let [existing (ports/get-task this id)]
    (when existing
      (let [deleted (task-core/mark-as-deleted existing)]
        (execute-one! ds ["UPDATE tasks SET deleted_at = ? WHERE id = ?"
                          (:deleted-at deleted) (str id)])))))

;; LIST excludes deleted records
(list-tasks [this params]
  (execute! ds ["SELECT * FROM tasks WHERE deleted_at IS NULL LIMIT ? OFFSET ?"
                (:limit params) (:offset params)]))
----

=== Custom Audit Log Interceptor

Interceptors handle cross-cutting concerns automatically. Define one that logs successful writes:

[source,clojure]
----
(defn audit-log-interceptor [action]
  {:name :audit-log
   :leave (fn [ctx]
            (let [user-id (get-in ctx [:request :identity :user-id])
                  status  (get-in ctx [:response :status])]
              (when (and user-id (<= 200 status 299))
                (log/info "AUDIT" {:action action :user-id user-id
                                   :at (java.time.Instant/now)}))
              ctx))})
----

Apply it to specific routes in `shell/http.clj`:

[source,clojure]
----
["/tasks" {:interceptors [auth-interceptor]}
 ["" {:post {:handler      create-handler
              :interceptors [(audit-log-interceptor :create-task)]}}]
 ["/:id" {:put    {:handler update-handler
                   :interceptors [(audit-log-interceptor :update-task)]}
          :delete {:handler delete-handler
                   :interceptors [(audit-log-interceptor :delete-task)]}}]]
----

    [:input {:name "title" :placeholder "New task..."}]

=== The Web Handler

[source,clojure]

(defn render-tasks-handler [request]

=== Property-Based Testing

[source,clojure]
----
(require '[clojure.test.check.generators :as gen]
         '[clojure.test.check.properties :as prop]
         '[clojure.test.check :as tc])

(def task-gen
  (gen/hash-map
   :id        gen/uuid
   :title     (gen/not-empty gen/string-alphanumeric)
   :priority  (gen/elements ["low" "medium" "high"])
   :completed (gen/return false)))

  (prop/for-all [task task-gen]
    (let [result (task-core/complete-task task)]
      (and (:completed result)
           (inst? (:completed-at result))
           (= (:id task) (:id result))))))

(tc/quick-check 100 complete-task-property)

== Chapter 14: Troubleshooting

*Q: I changed a file but the REPL doesn't see the change.* +
A: Run `(ig-repl/reset)`. This stops the system, reloads all changed namespaces, and restarts. For `defrecord` changes, use `(ig-repl/halt)` then `(ig-repl/go)`.

*Q: "Database is locked" (SQLite).* +
A: Multiple processes writing to the same `.db` file. Ensure only one REPL or server is running.

*Q: Migration failed midway.* +
A: Check the `migrations` table to see what was applied.

*Q: My token is rejected with "Expired".* +
A: Check your system clock. Ensure `JWT_SECRET` is at least 32 characters.

== Chapter 15: Glossary

[cols="1,3", options="header"]
|===
|Term |Definition

|*Functional Core (FC)*
|Pure functions and business logic with no side effects.

|*Imperative Shell (IS)*
|The layer that handles I/O: database, HTTP, logging.

|*Integrant*
|Micro-framework for data-driven system lifecycle management.

|*Aero*
|Configuration library for EDN files with environment-aware tags.

|*Malli*
|High-performance data validation and specification library.

|*Port*
|A Clojure Protocol defining an interface (the "what").

|*Adapter*
|A record implementing a Protocol for a specific technology.

|*Interceptor*
|A function that intercepts the request/response flow.

|*Uberjar*
|Single JAR file containing an application and all dependencies.

|*Hiccup*
|HTML represented as Clojure vectors and maps.

|*HTMX*
|Library for AJAX, CSS transitions, and WebSockets via HTML attributes.

== Chapter 16: Common Pitfalls

=== 1. Putting I/O in the Core

[source,clojure]
----
;; WRONG — core function calling the database
(ns boundary.task.core.task
  (:require [boundary.task.shell.persistence :as db]))  ; <-- never import shell

(defn create-if-valid [data]
  (if (valid? data)
    (db/save data)   ; <-- side effect in core
    (throw ...)))
----

Core should return data only. The Shell decides what to do with that data.

=== 2. snake_case in Clojure Code

[source,clojure]
----
;; WRONG — snake_case in Clojure
{:user_id "123" :first_name "John"}

;; CORRECT — always kebab-case internally
{:user-id "123" :first-name "John"}
----

Convert to `snake_case` only at the database boundary and `camelCase` only at the API boundary, using `boundary.shared.core.utils.case-conversion`.

=== 3. Ignoring the REPL

Don't restart the whole app to see changes. Keep your REPL connected, evaluate individual functions with Cmd+Enter (Calva), and use `(ig-repl/reset)` for configuration changes.

=== 4. Oversized Modules

If a module handles user management, storage, *and* notifications, it's doing too much. Follow the Single Responsibility Principle — different domain entities belong in separate modules.

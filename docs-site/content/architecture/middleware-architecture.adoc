= Middleware Architecture and Grouping
:revdate: 2026-02-24

[abstract]
--
Boundary applies middleware in layers so each route group gets the right behavior with predictable ordering.
--

== Overview

HTTP middleware is applied in three scopes:

1. Handler-level (outside router)
2. Router-level (all routes)
3. Route-group-level (API/web/root differences)

[source,text]
----
Request
  -> handler-level middleware
  -> router-level middleware
  -> route-group middleware
  -> route handler
Response (reverse order)
----

== 1. Handler-Level Middleware

Purpose:

* concerns that should run before routing
* static resource serving

Typical behavior:

* if static file exists, return directly
* otherwise continue into router

== 2. Router-Level Middleware

Purpose:

* behavior shared by every route

Typical responsibilities:

* top-level exception translation
* request/response logging
* correlation ID propagation

Rule: keep this set small and universal.

== 3. Route-Group Middleware

Purpose:

* apply interface-specific behavior

=== API Group

Common stack:

* content negotiation
* parameter parsing
* request/response coercion

Used for JSON/Transit-style endpoints.

=== Web Group

Common stack:

* HTML/HTMX handling
* session and CSRF (when enabled)
* multipart support (when needed)

Used for browser-facing routes.

=== Root/Infra Group

Common stack:

* minimal middleware only

Used for health/docs/static-adjacent endpoints.

== Ordering Rules

Middleware is nested. The outermost layer handles the broadest concerns.

Recommended order (outer -> inner):

1. exception handling
2. correlation/logging
3. authn/authz (if global)
4. format/params/coercion
5. route handler

Why:

* outer exception middleware catches downstream failures
* logging can measure full request duration
* coercion runs before business logic

== Error Handling Contract

Middleware should convert failures to uniform boundary responses.

* include a stable error type
* include correlation ID
* avoid leaking internal stack traces in production

== Adding New Middleware Safely

1. Decide scope (handler, router, or route-group).
2. Document ordering requirements.
3. Add tests for success and failure paths.
4. Verify interaction with existing middleware.
5. Confirm no behavior regressions for unrelated route groups.

== Testing Strategy

=== Unit

Test one middleware in isolation:

* request pass-through
* response transformation
* exception behavior

=== Integration

Test full stack for each route group:

* API: coercion + error format
* Web: session/CSRF behavior
* Root: minimal overhead behavior

=== Ordering

Use marker middleware in tests to assert execution order.

== Common Pitfalls

* Putting static resource middleware inside router stack
* Applying API coercion middleware to web HTML routes
* Duplicating `/api` prefixes across module and system routing
* Catching exceptions too deep (missing outer safety net)

= Component Architecture

[abstract]
--
This page describes Boundary's concrete component boundaries and where each responsibility lives in the codebase.
--

== Audience and Scope

* Audience: module authors and maintainers
* Scope: component responsibilities, interaction flow, and code locations
* Prerequisite: link:overview.adoc[Architecture Overview]

== Component Types

=== Functional Core Components

Purpose: pure domain decisions and data transformations.

Rules:

* No I/O, no logging, no DB calls
* No runtime mutation or hidden state
* Depends on data and ports only

Typical location:

[source,text]
----
libs/{module}/src/boundary/{module}/core/
----

=== Port Components

Purpose: define contracts between core logic and infrastructure.

Rules:

* Protocols/interfaces only
* No implementation details
* Stable names and semantics

Typical location:

[source,text]
----
libs/{module}/src/boundary/{module}/ports.clj
----

=== Shell Components

Purpose: execute side effects and orchestrate interfaces.

Responsibilities:

* Input validation/coercion
* Calling core functions
* Executing adapter operations
* Mapping errors to interface responses
* Cross-cutting concerns (logging, metrics, tracing)

Typical location:

[source,text]
----
libs/{module}/src/boundary/{module}/shell/
----

== Platform Components

`libs/platform` provides shared runtime infrastructure:

* HTTP interface and middleware
* Database adapters and migrations
* CLI entry points and command middleware
* System/module wiring

Key locations:

[source,text]
----
libs/platform/src/boundary/platform/shell/interfaces/http/
libs/platform/src/boundary/platform/shell/interfaces/cli/
libs/platform/src/boundary/platform/shell/adapters/database/
libs/platform/src/boundary/platform/shell/system/wiring.clj
----

== Observability Components

`libs/observability` provides shared telemetry components:

* Logging (`boundary.observability.logging.*`)
* Metrics (`boundary.observability.metrics.*`)
* Error reporting (`boundary.observability.errors.*`)

Adapter implementations live under `shell.adapters` namespaces.

== Interaction Flow

[source,text]
----
HTTP/CLI/Web request
  -> shell handler/service
  -> validation + normalization
  -> core function(s)
  -> port calls (via injected implementations)
  -> adapter side effects (DB, queue, external API)
  -> normalized response/error
----

== Module Boundaries

Boundary modules communicate through explicit contracts, not direct internal imports.

Allowed:

* Import another module's public ports/schema when required
* Use shared libraries (`core`, `platform`, `observability`)

Avoid:

* Importing another module's `shell` internals
* Importing another module's private helper namespaces
* Re-implementing shared utilities already in `boundary.core.utils.*`

== Example Map

[source,text]
----
libs/user/src/boundary/user/
├── core/               # auth/session decisions
├── ports.clj           # user service/repository protocols
├── schema.clj          # malli schemas
└── shell/
    ├── service.clj     # orchestration
    ├── persistence.clj # DB adapter integration
    ├── http.clj        # HTTP handlers/routes
    └── module_wiring.clj
----

== Concrete Examples

=== Core Function (Pure)

[source,clojure]
----
(ns myapp.user.core.logic)

(defn prepare-user-create
  [input now]
  (-> input
      (assoc :id (random-uuid))
      (assoc :created-at now)
      (assoc :updated-at now)))
----

=== Shell Service (Orchestration + I/O)

[source,clojure]
----
(ns myapp.user.shell.service
  (:require [myapp.user.core.logic :as logic]
            [boundary.core.validation :as validation]))

(defn create-user
  [repo schema input now]
  (validation/validate! schema input)
  (let [entity (logic/prepare-user-create input now)]
    (repo/save-user! repo entity)))
----

== Verification Checklist

When reviewing module changes:

1. Core namespace imports only core/ports/shared pure utilities.
2. Validation and I/O occur in shell, not core.
3. New side effects are behind a port or adapter boundary.
4. Error ex-data includes `:type` for boundary-safe translation.
5. Public docs/examples reference `libs/...` paths and current namespaces.

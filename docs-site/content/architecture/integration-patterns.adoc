= Integration Patterns

[abstract]
--
Boundary exposes consistent behavior across HTTP, CLI, and web interfaces by reusing the same shell services, core functions, schemas, and error model.
--

== Audience and Scope

* Audience: API integrators, platform engineers, module maintainers
* Scope: interface patterns, consistency rules, versioning, and backward compatibility
* Prerequisites: link:overview.adoc[Architecture Overview], link:components.adoc[Component Architecture]

== Principle

Different interfaces, same business behavior.

[source,text]
----
Interface (HTTP/CLI/Web)
  -> Shell handler
  -> Validation/coercion
  -> Core decision logic
  -> Port-driven side effects
  -> Normalized result/error
----

== Interface Patterns

=== HTTP Pattern

Used for REST APIs and service integrations.

Responsibilities:

* Parse request and coerce params/body
* Invoke shell service
* Map result to status/body/headers
* Return problem details for typed errors

Typical locations:

[source,text]
----
libs/platform/src/boundary/platform/shell/interfaces/http/
libs/{module}/src/boundary/{module}/shell/http*.clj
----

Minimal handler shape:

[source,clojure]
----
(ns myapp.http
  (:require [boundary.user.shell.service :as user-service]))

(defn create-user-handler [svc]
  (fn [request]
    (let [user-data (get-in request [:parameters :body])
          result (user-service/create-user svc user-data)]
      {:status 201
       :body result})))
----

=== CLI Pattern

Used for operations and automation.

Responsibilities:

* Parse command/options
* Reuse same shell services as HTTP
* Map typed errors to exit codes
* Support human and JSON output

Typical locations:

[source,text]
----
libs/platform/src/boundary/platform/shell/interfaces/cli/
libs/{module}/src/boundary/{module}/shell/cli*.clj
----

Minimal command shape:

[source,clojure]
----
(ns myapp.cli
  (:require [boundary.user.shell.service :as user-service]))

(defn create-user-command [svc opts]
  (try
    (let [result (user-service/create-user svc opts)]
      {:exit-code 0 :output result})
    (catch Exception e
      {:exit-code 1
       :output {:error (.getMessage e)}})))
----

=== Web Pattern

Used for browser-facing flows (server-rendered pages, HTMX, real-time updates).

Responsibilities:

* Adapt request/response format for HTML/partials/events
* Reuse the same shell service and schemas
* Keep interface-specific rendering in shell/web handlers

Typical locations:

[source,text]
----
libs/admin/src/boundary/admin/shell/http.clj
libs/platform/src/boundary/platform/shell/interfaces/web/
----

== Consistency Contract

Across interfaces, keep these invariant:

1. Same input schema semantics.
2. Same core decision logic.
3. Same typed error categories.
4. Same domain side effects (through ports).

Interface-specific differences should only be transport-level:

* HTTP status and headers
* CLI exit codes and formatting
* HTML/JSON/event rendering

== Error Mapping

Boundary expects typed ex-data (`:type`) so each interface can map consistently.

Common mappings:

[cols="1,1,1"]
|===
|Error Type |HTTP |CLI

|`:validation-error`
|400
|2

|`:unauthorized`
|401
|3

|`:forbidden`
|403
|4

|`:not-found`
|404
|5

|`:conflict`
|409
|6

|`:internal-error`
|500
|1
|===

== Versioning Pattern

Use explicit API versions at the interface edge while keeping internal contracts stable.

Recommendations:

* Version routes (`/api/v1`, `/api/v2`) in platform HTTP routing.
* Keep shell/core signatures stable where possible.
* Put compatibility shims in interface layer, not in core logic.
* Publish deprecation windows before removing old versions.

== Integration Checklist

Before shipping a new endpoint/command/page:

1. Schema exists and validates expected payload.
2. Shell orchestrates; core remains pure.
3. Errors include `:type` in ex-data.
4. Interface mapping for success/error is covered by tests.
5. Observability hooks (logging/metrics/errors) are present.

= Observability Integration Guide

[abstract]
--
This guide explains how to add logging, metrics, and error reporting in Boundary without mixing observability concerns into core business logic.
--

== Audience and Scope

* Audience: backend engineers and module maintainers
* Scope: integration approach, wiring, environments, and testing
* Prerequisite: link:error-handling-observability[Error Handling & Observability]

== Core Principle

Observability belongs in shell and platform layers, not in core logic.

[source,text]
----
Interface -> shell service -> core logic -> ports/adapters
                     |
                     +-> logging / metrics / error reporting
----

== Recommended Integration Pattern

Use interceptor wrappers provided by platform:

* Service operations: `execute-service-operation`
* Persistence operations: `execute-persistence-operation`

Benefits:

* Removes repeated try/catch/log boilerplate
* Ensures consistent breadcrumbs and error context
* Keeps domain logic readable and testable

== Wiring Requirements

A shell component typically receives:

* `:logger`
* `:metrics`
* `:error-reporter`
* domain dependencies (repository/service clients)

Example Integrant-style dependency map:

[source,clojure]
----
{:my/module-service
 {:logger #ig/ref :boundary/logging
  :metrics #ig/ref :boundary/metrics
  :error-reporter #ig/ref :boundary/error-reporting
  :repository #ig/ref :my/module-repository}}
----

== Where to Instrument

=== Service Layer

Track:

* operation start/end
* duration
* business failure categories

Avoid:

* logging raw secrets/PII
* encoding transport-specific concerns in core

=== Persistence Layer

Track:

* query/operation type
* table/resource name
* success/failure counts
* latency

Avoid:

* leaking SQL internals in user-facing errors

=== Interface Layer (HTTP/CLI/Web)

Track:

* request/command identity
* correlation ID
* status/exit code
* duration

Map all exceptions into typed errors (`:type` in ex-data).

== Environment Strategy

=== Development

* readable logging (`stdout`)
* relaxed sampling
* optional no-op metrics/error reporting

=== Staging

* structured logs
* metrics enabled
* error reporting enabled with higher sampling

=== Production

* structured logs only
* metrics and dashboards enabled
* error reporting with redaction and lower sampling

== Data Hygiene

Always redact or omit sensitive data:

* tokens
* passwords
* secrets
* personal data not required for debugging

Prefer event fields like `:user-id` or `:tenant-id` over raw payload dumps.

== Testing Observability

Use capturing test doubles for logger/metrics/reporter and assert:

1. expected events emitted for success path
2. expected events emitted for failure path
3. correlation/context fields are present
4. no sensitive fields are leaked

== Integration Checklist

1. Core remains pure (no logging/metrics/error calls).
2. Shell uses interceptor wrappers for service/persistence operations.
3. Exceptions include `:type` in ex-data.
4. Correlation ID is propagated across layers.
5. Production config includes redaction and sampling controls.

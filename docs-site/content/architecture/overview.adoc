= Architecture Overview

[abstract]
--
Boundary uses a Functional Core / Imperative Shell architecture inside module-based libraries. This page gives the shortest accurate map of how code is organized and how dependencies flow.
--

== Audience and Scope

* Audience: domain developers, platform engineers, API integrators
* Scope: architecture model, dependency rules, module layout, and cross-module boundaries

== Core Model

Boundary separates code into three concerns:

* Functional core: pure business logic (`core/`)
* Ports: protocols and contracts (`ports.clj`)
* Imperative shell: I/O, validation, adapters, interfaces (`shell/`)

[source,text]
----
Shell (I/O, validation, adapters)
        |
        v
Core (pure decisions and transformations)
        |
        v
Ports (abstractions/protocols)
----

== Dependency Rules

[cols="2,1,3"]
|===
|Direction |Status |Rule

|Shell -> Core
|Allowed
|Shell orchestrates pure core logic.

|Core -> Ports
|Allowed
|Core depends on abstractions only.

|Shell -> Adapters
|Allowed
|Shell wires concrete implementations.

|Core -> Shell
|Forbidden
|Core never depends on I/O or infrastructure.

|Core -> Adapters
|Forbidden
|Core never imports concrete implementations.
|===

== Module Layout

Boundary is a monorepo with library modules under `libs/`:

[source,text]
----
libs/
├── core/          # Shared validation, interceptors, utilities
├── observability/ # Logging, metrics, error reporting
├── platform/      # HTTP, DB, CLI, system wiring
├── user/          # Authentication, authorization, MFA
├── admin/         # Admin UI and CRUD
├── storage/       # Local/S3 file storage and image support
├── cache/         # Cache abstraction and adapters
├── jobs/          # Background jobs and retries
├── email/         # Email delivery
├── realtime/      # WebSocket/SSE messaging
├── tenant/        # Multi-tenancy lifecycle
├── scaffolder/    # Module generator
└── external/      # External adapters (in development)
----

Each module follows the same structure:

[source,text]
----
libs/{module}/src/boundary/{module}/
├── core/
├── shell/
├── ports.clj
└── schema.clj
----

== Runtime Composition

At runtime, `platform` composes modules via Integrant wiring. Interfaces (HTTP/CLI/Web) call shell services, shells call core, and cores operate on validated data and ports.

== Architecture Guarantees

* Testability: core functions are unit-testable without infrastructure.
* Consistency: the same core logic is reused across HTTP/CLI/Web entry points.
* Replaceability: adapters can change (DB, queue, providers) without rewriting core logic.
* Modularity: domains evolve in separate libraries with explicit contracts.

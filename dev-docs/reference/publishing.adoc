= Publishing Libraries

This guide covers everything needed to publish Boundary libraries to Clojars — locally or via GitHub Actions.

---

== Prerequisites

=== Clojars account

Create an account at https://clojars.org[clojars.org] if you don't have one.

=== Deploy token

Clojars requires a deploy token — **not** your account password.

. Log in at https://clojars.org
. Go to *Profile → Deploy Tokens*
. Click *Generate Token*, name it (e.g. `boundary-deploy`)
. Copy the token immediately — it is shown only once

The token looks like: `CLOJARS_XXXXXXXXXXXXXXXXXXXXXXXX`

=== Local environment variables

[source,bash]
----
export CLOJARS_USERNAME=your-clojars-username   # username, not email
export CLOJARS_PASSWORD=your-deploy-token        # token from above
----

Add these to your shell profile (`~/.zshrc` or `~/.bashrc`) for convenience.

== Version management

Each publishable artifact has a `(def version "…")` in its own `build.clj`.
This includes the libraries under `libs/*` and `boundary-tools/`.

The Git tag does not define the artifact versions.  It only triggers the
workflow.  Clojars versions are read from each artifact's `build.clj` at
publish time.

=== Bumping the version

[source,bash]
----
# From the repository root — update all library build.clj files
sed -i '' 's/(def version "OLD")/(def version "NEW")/' libs/*/build.clj

# boundary-tools is versioned separately and must be bumped too for a full tagged release
sed -i '' 's/(def version "OLD")/(def version "NEW")/' boundary-tools/build.clj

# Verify
grep "^(def version" libs/*/build.clj
grep "^(def version" boundary-tools/build.clj
----

=== Version format

Boundary follows semantic versioning:

* `1.0.0-alpha-N` — alpha release, N is a sequential counter
* `1.0.0-beta-N` — beta release
* `1.0.0` — stable release
* `1.0.1` — patch release

[IMPORTANT]
====
Clojars does **not** allow overwriting an existing version.
Always bump the version before deploying — you cannot re-deploy the same version string.
====

== Deploying locally

=== All 21 artifacts in one command

[source,bash]
----
bb deploy --all
----

This deploys all publishable artifacts, including `boundary-tools`, in dependency
order with a 30-second pause between artifacts to allow Clojars indexing.

=== Only libraries not yet on Clojars

[source,bash]
----
bb deploy --missing
----

Useful when resuming a partially failed deploy run.

=== Specific libraries

[source,bash]
----
bb deploy core platform user
----

=== Single library manually

[source,bash]
----
cd libs/core
clojure -T:build clean
clojure -T:build deploy
----

=== Verify publication

After deploying, check that the artifacts are visible on Clojars:

[source,bash]
----
# Quick check — expects HTTP 200 for each library
bb deploy --missing   # libraries not yet published will be listed
----

Or open in a browser: https://clojars.org/org.boundary-app/boundary-core

== Deploying via GitHub Actions

The workflow is defined in `.github/workflows/publish.yml`.

=== One-time setup: GitHub secrets

Add the following secrets to the GitHub repository:

*GitHub → Settings → Secrets and variables → Actions → New repository secret*

[cols="1,2",options="header"]
|===
|Secret name |Value
|`CLOJARS_USERNAME` |Your Clojars username (not email)
|`CLOJARS_PASSWORD` |Your Clojars deploy token (from clojars.org/tokens)
|===

=== Trigger: push a version tag

This is the normal release flow:

[source,bash]
----
# 1. Bump versions in all build.clj files
sed -i '' 's/(def version "1.0.0-alpha-3")/(def version "1.0.1-alpha-4")/' libs/*/build.clj
sed -i '' 's/(def version "1.0.0-alpha")/(def version "1.0.1-alpha-4")/' boundary-tools/build.clj

# 2. Commit and tag
git add libs/*/build.clj boundary-tools/build.clj
git commit -m "Bump version to 1.0.1-alpha-4"
git tag -a v1.0.1-alpha-4 -m "Release 1.0.1-alpha-4"
git push origin main
git push origin v1.0.1-alpha-4
----

GitHub Actions will:

. Build the Tailwind CSS bundle (required for the `ui-style` jar)
. Run `bb deploy --all` with the repository secrets as credentials
. Create a GitHub Release marked as pre-release for alpha/beta versions

=== Trigger: manual dispatch

For deploying without a tag (e.g. re-running after a partial failure):

. Go to *GitHub → Actions → Publish to Clojars*
. Click *Run workflow*
. Optionally enter a version label for the release notes

== What the workflow does

[source,yaml]
----
- Setup Clojure + Babashka       # DeLaGuardo/setup-clojure with bb: latest
- Cache ~/.m2 and ~/.gitlibs     # Speeds up subsequent runs
- npm ci + npm run build:css:admin  # Builds Tailwind CSS for ui-style jar
- bb deploy --all                # Deploys all 21 artifacts in order, 30s between each
- Create GitHub Release          # Only on tag pushes; pre-release for alpha/beta
----

Total CI time is approximately 15–20 minutes (21 artifacts × 30s indexing pause + build time).

== Release checklist

Before pushing a release tag:

* [ ] All tests pass: `clojure -M:test:db/h2`
* [ ] No lint errors: `clojure -M:clj-kondo --lint src test libs/*/src libs/*/test`
* [ ] `CHANGELOG.md` updated
* [ ] Version bumped in all relevant `build.clj` files, including `boundary-tools/build.clj` for a full tagged release
* [ ] Changes committed and pushed to `main`
* [ ] Local deploy works: `bb deploy core` (quick smoke check)

== Troubleshooting

=== 401 Unauthorized

[source]
----
authentication failed for https://clojars.org/repo/..., status: 401 Unauthorized
----

Work through these causes in order:

==== 1. Account password used instead of deploy token _(most common)_

`CLOJARS_PASSWORD` must be a deploy token, not your login password.
See <<_prerequisites>> for how to generate one.

==== 2. GitHub secret missing or empty

The CI log masks any set value as `***` — even an empty string.
Verify both secrets exist and are non-empty:
*GitHub → Settings → Secrets and variables → Actions*

==== 3. Group `org.boundary-app` not verified

Clojars requires proof of domain ownership for groups outside `io.github.*` / `com.github.*`.
For `org.boundary-app`, add a DNS TXT record for `boundary-app.org` as described at
https://github.com/clojars/clojars-web/wiki/Verified-Group-Names[Clojars: Verified Group Names].

=== Version already exists

[source]
----
Error: version 1.0.0-alpha already exists on clojars
----

Bump the version in all `build.clj` files and redeploy.
Deleting an existing version on Clojars is only possible within 24 hours of upload.

=== Dependency not found after deploy

[source]
----
Could not find artifact org.boundary-app/boundary-core:jar:1.0.1-alpha-4
----

. Check that the library was actually published: open `https://clojars.org/org.boundary-app/boundary-core`
. Wait 1–2 minutes for Clojars to finish indexing
. Clear your local Maven cache: +
  `rm -rf ~/.m2/repository/org/boundary-app/`

=== CSS not included in ui-style jar

The compiled Tailwind CSS (`daisy-admin.css`) is gitignored and must be built before packaging.

Locally:
[source,bash]
----
npm ci && npm run build:css:admin
cd libs/ui-style && clojure -T:build deploy
----

In CI this is handled automatically by the workflow's *Build shared UI CSS bundle* step.

=== Inspecting a generated POM

[source,bash]
----
cd libs/core
clojure -T:build clean && clojure -T:build jar
cat target/classes/META-INF/maven/org.boundary-app/boundary-core/pom.xml
----

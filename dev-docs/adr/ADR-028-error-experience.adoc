= ADR-028: Error Experience & Auto-Recovery
:status: Proposed
:date: 2026-04-18
:deciders: boundary-team

== Status

Proposed

== Context

=== Background

ADR-022 established canonical error types with HTTP status code mapping and
the `enforce-error-type?` diagnostic. The validation system in `libs/core`
provides structured error results with "Did you mean?" suggestions. These
foundations are solid but developer-facing.

=== Problem

* Errors tell you *what* went wrong but not *how to fix it*.
* No error code system -- errors are identified by type keyword and message
  text, making them hard to look up in documentation.
* Stack traces in development show full JVM frames, burying the relevant
  Boundary namespace under framework and Java internals.
* Common recoverable errors (missing migration, missing env var, unintegrated
  module) require developers to know the fix command.

== Decision

=== Error code system: BND-xxx

Every Boundary error gets a structured code:

[cols="1,3"]
|===
| Range | Category

| `BND-1xx`
| Configuration errors (missing env vars, invalid providers, bad config)

| `BND-2xx`
| Validation errors (schema failures, invalid input)

| `BND-3xx`
| Persistence errors (missing tables, migration issues, connection failures)

| `BND-4xx`
| Authentication/authorization errors

| `BND-5xx`
| Interceptor pipeline errors

| `BND-6xx`
| FC/IS boundary violations
|===

Each code maps to documentation with: what it means, common causes, how to
fix it, and a code example.

=== Rich error output in dev mode

Dev-mode errors include:

* Error code and title
* Affected handler/schema/namespace
* Structured error details with field-level breakdown
* Input received vs. expected shape (with generated example)
* Fix suggestion with specific command or code change
* Link to dashboard error page and documentation

=== Auto-recovery: `(fix! *e)`

A REPL function that analyzes the last exception and offers automatic fixes:

[cols="1,2"]
|===
| Error | `(fix! \*e)` action

| Missing migration
| Runs `migrate up`

| Missing env var
| Sets for current session

| Missing module wiring
| Runs `integrate!`

| Invalid config value
| Suggests correct value, applies if unambiguous

| Missing dependency
| Suggests addition to `deps.edn` (always interactive, requires confirmation)

| FC/IS violation
| Shows refactoring steps + offers `(ai/refactor-fcis ...)`
|===

=== Stack trace filtering

In dev mode, stack traces highlight user code and dim framework/JVM frames.
The relevant Boundary namespace appears at the top.

== Consequences

=== Positive

* *Errors as conversations* -- every error tells you what to do next.
* *Faster recovery* -- `(fix! *e)` resolves common issues in one command.
* *Searchable errors* -- BND-xxx codes make errors easy to look up and discuss.
* *Readable stack traces* -- user code first, noise dimmed.

=== Negative / risks

* *Error catalog maintenance* -- new error codes must be added as the
  framework grows.
* *`(fix! *e)` safety* -- auto-fixes must not silently break things. The
  `deps.edn` modification case is especially sensitive (version conflicts).
* *Rich output verbosity* -- detailed error output may be too verbose for
  some developers.

=== Mitigation

* Error codes are defined in `boundary.devtools.core.error_codes` with tests
  ensuring all codes have documentation.
* `(fix! *e)` always explains what it will do before doing it. Destructive
  fixes (deps.edn modification) require explicit confirmation.
* Rich output respects guidance level: `:full` shows everything, `:minimal`
  shows code + one-line fix, `:off` shows standard Clojure error.

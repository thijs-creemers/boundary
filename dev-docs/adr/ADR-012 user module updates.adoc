= ADR-012: User Module Admin Integration — Split-Table Architecture
:status: Accepted
:date: 2026-03-22
:deciders: boundary-team

== Status

Accepted

== Context

The `boundary-user` library stores user data across two tables:

* `auth_users` — authentication fields: `id`, `email`, `active`, `mfa_enabled`, `mfa_secret`, `password_hash`, `created_at`, `updated_at`, `deleted_at`
* `users` — profile fields: `id`, `name`, `role`, `tenant_id`, `avatar_url`, `theme`, `language`, `timezone`, `date_format`, `time_format`, `notifications_email`, `notifications_push`, `notifications_sms`, `login_count`, `last_login`

Three compounding bugs were discovered during admin UI integration:

=== Bug 1 — boundary-platform: silent exception swallowing

In `persistence-error-handling`, the `:error` handler added `:error-info` to the
interceptor context but did NOT preserve the `:error` key.  `execute-persistence-operation`
checks `(:error result-context)` to decide whether to rethrow; without it, any SQL
exception (wrong column name, type mismatch, constraint violation) was silently discarded
and `nil` returned.  Every `update-entity`, `create-entity`, and `delete-entity` call
system-wide was affected.

=== Bug 2 — boundary-admin: update handler discards result

`update-entity-handler` bound the `ports/update-entity` return value as `_updated-entity`
(discarded), then immediately called `ports/get-entity` again for a fresh fetch.  This meant
a failed (nil-returning) update would still render a success toast with stale data.

=== Bug 3 — boundary-user config: missing split-table admin wiring

The admin entity config for users (`resources/conf/dev/admin/users.edn`) had no `:table-name`
key, so the admin service issued `UPDATE nil SET …` — a SQL error silently swallowed by
Bug 1.  Additionally, `:email` and `:active` (which live in `auth_users`) were included in
editable field-groups without being listed as `:readonly-fields`, causing a second SQL error
(`column email not found in table users`).

== Decision

=== Fix 1 — Preserve `:error` key in `persistence-error-handling`

`libs/platform/src/boundary/platform/shell/persistence_interceptors.clj`

The `:error` handler now includes `:error exception` in the `assoc` call alongside
`:error-info`, so `execute-persistence-operation` correctly rethrows instead of returning `nil`.

Unit tests were added in
`libs/platform/test/boundary/platform/shell/persistence_interceptors_test.clj` covering:

* `:error` handler sets the `:error` key (both `ExceptionInfo` and plain `Exception`)
* `execute-persistence-operation` rethrows when `db-logic-fn` throws
* Happy path: correct result is returned when `db-logic-fn` succeeds

=== Fix 2 — Use `ports/update-entity` result directly

`libs/admin/src/boundary/admin/shell/http.clj`

Removed the `_updated-entity` binding and the redundant `ports/get-entity` round-trip.
`ports/update-entity` already fetches the updated record internally (join-aware SELECT) and
returns it.  The handler now uses this value directly as `updated-record`, eliminating one
unnecessary database query and ensuring a failed update propagates as an exception (via Fix 1)
rather than rendering a false success toast.

=== Fix 3 — Correct split-table admin entity config

`resources/conf/dev/admin/users.edn`

Three additions:

1. **`:table-name :users`** — the admin service uses this key as the `UPDATE` target.
   All editable profile fields live in `users`, so this is the correct table.

2. **`:email` and `:active` added to `:readonly-fields`** — these fields live in `auth_users`.
   The generic admin `UPDATE users SET email = …` would fail.  Email changes require a
   dedicated auth flow; activation toggling should be a dedicated action.

3. **`:mfa-enabled` added to `:readonly-fields`** — also lives in `auth_users`.  Displayed
   via `:query-overrides :select` for read-only visibility; must not appear in the UPDATE set.

Fields remain in `:field-groups` for display purposes; since they are now in `:readonly-fields`
they render as read-only labels, not form inputs.

== Decision (continued)

=== Fix 4 — Generic split-table update support

With Fixes 1–3 in place, `:email` and `:active` were rendered read-only in the admin UI.
The correct UX is a single edit form that transparently routes fields to the right table.

==== Design

An optional `:split-table-update` key was added to entity configs:

[source,edn]
----
:split-table-update {:secondary-table :auth_users
                     :secondary-fields #{:email :active}}
----

When `update-entity` in `service.clj` sees this key it:

1. Partitions `sanitized-data` into `primary-data` (not in `:secondary-fields`) and
   `secondary-data` (only secondary fields).
2. Adds `updated-at` to each half independently when `:updated-at` exists in
   `:fields` (same guard used on the single-table path).
3. Runs both UPDATEs inside `db/with-transaction*` so they commit or roll back atomically.
4. Skips an UPDATE for a half that has no fields after sanitisation (e.g. form only
   touched profile fields → no `auth_users` UPDATE issued).
5. The join-aware SELECT that returns the merged record is unchanged.

For entities without `:split-table-update` the code path is identical to before.

==== Files changed

* `libs/admin/src/boundary/admin/shell/service.clj` — `split-update-data` private helper;
  `update-entity` branched on `:split-table-update`.
* `resources/conf/dev/admin/users.edn` — `:email` and `:active` removed from
  `:readonly-fields`; `:split-table-update` added.  `:mfa-enabled` remains read-only
  (controlled by MFA enrollment flow, not the generic edit form).
* `libs/admin/test/boundary/admin/shell/split_table_update_test.clj` — NEW integration
  tests covering both-tables update, primary-only, secondary-only, and single-table
  regression guard.

== Consequences

* SQL exceptions from persistence operations now propagate correctly system-wide instead of
  being silently swallowed.  Applications that relied on nil-return-on-error (intentionally
  or accidentally) may now see exceptions where they previously saw nil; callers should be
  reviewed if they handle nil returns from `update-entity`, `create-entity`, or `delete-entity`.

* `:email` and `:active` are now editable in the admin user form.  `:mfa-enabled` remains
  read-only — it is controlled by the dedicated MFA enrollment flow.

* Any future split-table entity can opt into transactional multi-table updates with a single
  `:split-table-update` config key; no service code changes needed.

* One fewer database query is issued per successful admin update (removed the redundant
  `get-entity` round-trip after `update-entity`).

= ADR-018: Admin Module — :belongs-to Widget for Foreign Key Lookups
:status: Proposed
:date: 2026-03-26
:deciders: boundary-team

== Status

Proposed

== Context

The admin module currently has no native support for foreign key lookups. When an
entity contains a UUID column that references another entity (e.g.
`contractor.tenant_id → tenants.id`), the admin UI can only display the raw UUID —
not the human-readable name of the related entity.

The current workaround is `:query-overrides :join` in the entity-config EDN:

[source,edn]
----
:query-overrides
{:select [:contractors.* [:public.tenants/name :tenant-name]]
 :join   [[:public.tenants]
          [:= [:cast :contractors/tenant_id :uuid] :public.tenants/id]]}
----

This has three problems:

1. *Cross-schema joins* — in schema-per-tenant setups, related entities may live
   in a different PostgreSQL schema (`public.tenants` vs. `tenant_abc.contractors`).
   The EDN config must then contain schema-qualified table names, coupling the
   config to the database layout.

2. *Test incompatibility* — H2 (the in-memory test database) does not support
   PostgreSQL-specific cast syntax (`::uuid`). Any join with type conversion
   breaks the test suite unless explicitly handled.

3. *Maintainability* — HoneySQL join syntax in EDN files is error-prone,
   not validatable by the schema provider, and difficult to read for
   non-Clojure developers managing entity configurations.

=== Concrete trigger

During the implementation of the ZZPGuard admin configuration for `contractors`,
the requirement arose to display the `tenant-name` alongside the `tenant-id` (UUID)
in the detail view. This was not possible without the workaround above.

== Decision

The admin module will add support for a new field type and widget: `:belongs-to`.

=== EDN configuration (application side)

[source,edn]
----
:tenant-name
{:type        :belongs-to
 :widget      :belongs-to
 :label       "Tenant"
 :readonly    true
 :relation    {:entity        :tenants
               :foreign-key   :tenant-id
               :display-field :name}}
----

=== Behaviour

* When fetching an entity (detail and list views), the service resolves
  `:belongs-to` fields via an additional query or JOIN, depending on the
  database adapter.
* The value displayed in the UI is the value of `:display-field` on the
  related entity.
* The field is always read-only (writing through a foreign key lookup is not
  supported in v1).
* In the list view the display value is shown; sorting on this field is
  optional and configurable per entity config.

=== Implementation approach

==== Schema (`boundary.admin.schema`)

* `FieldType` gains `:belongs-to` as a new value.
* `FieldWidget` gains `:belongs-to` as a new value.
* New optional key `:relation` on `FieldConfig`:
+
[source,clojure]
----
[:relation {:optional true}
 [:map
  [:entity        :keyword]   ; name of the related entity
  [:foreign-key   :keyword]   ; field on the current entity (the FK)
  [:display-field :keyword]]] ; field on the related entity to display
----

==== Service (`boundary.admin.shell.service`)

* `list-entities` and `get-entity` detect `:belongs-to` fields in the
  entity config and append the resolved values to each record before returning.
* The lookup is a separate `get-entity` call per unique FK value
  (batching as an optimisation in a later iteration).
* Cross-schema lookups are handled by the schema provider, which knows the
  table and schema name of the related entity.

==== UI (`boundary.admin.core.ui`)

* The `:belongs-to` widget renders the field as read-only text (identical to
  a `:text-input` with `readonly: true`).
* The field is not editable in forms.

=== Out of scope (v1)

* Editable belongs-to (e.g. a dropdown to change the FK value).
* Concatenating multiple display fields (e.g. `first-name + last-name`).
* Sorting on belongs-to columns in the list view.
* Automatic batching of lookups.

== Consequences

=== Positive

* No raw SQL or HoneySQL in EDN configurations for simple FK lookups.
* Works with all database adapters (H2, PostgreSQL) without adapter-specific
  cast syntax.
* Consistent with the existing `:has-many` relation configuration in the
  admin module.
* EDN config remains declarative and validatable via Malli.

=== Negative / risks

* Extra database query per `:belongs-to` field per row (N+1 risk on large
  lists). Mitigation: batching in a later iteration, or an optional JOIN
  strategy per entity config.
* The schema provider must know the table location of related entities
  (cross-schema). This requires all entities to be in the allowlist, or for
  the schema provider to have a fallback for unknown entities.

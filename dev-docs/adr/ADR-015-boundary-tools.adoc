= ADR-015: boundary-tools — Distributing Developer Tooling to Starter Projects
:status: Accepted
:date: 2026-03-23
:deciders: boundary-team

== Status

Accepted — implemented 2026-03-23

== Context

The boundary monorepo ships a growing set of Babashka developer tools that
significantly improve the development experience:

[cols="1,3", options="header"]
|===
| Command | Purpose

| `bb scaffold`
| Interactive module scaffolding wizard (generate/new/field/endpoint/adapter)

| `bb scaffold ai`
| Natural-language module scaffolding via AI

| `bb ai explain`
| Explain a Clojure/Boundary stacktrace via AI

| `bb ai gen-tests`
| Generate a test namespace for a source file

| `bb ai sql`
| Generate HoneySQL from a natural-language description

| `bb ai docs`
| Generate `AGENTS.md` for a module

| `bb i18n find`
| Locate a translation key by string or grep all usages of a key (ADR-013)

| `bb i18n scan`
| Find unexternalised string literals in `core/ui.clj` files (ADR-013)

| `bb i18n missing`
| Report translation keys absent in one or more locale files (ADR-013)

| `bb i18n unused`
| Report catalogue keys that are not referenced in the codebase (ADR-013)

| `bb create-admin`
| Create the first admin user for a new project

| `bb check-links`
| Validate local markdown links in `AGENTS.md` files

| `bb smoke-check`
| Verify `deps.edn` aliases and key tool entrypoints

| `bb install-hooks`
| Configure git hooks path to `.githooks`

| `bb deploy`
| Publish one or more boundary libraries to Clojars
|===

All of these tools live in the `boundary` monorepo (`scripts/`, `bb.edn`) and
are not available to projects that start from `boundary-starter`.  A developer
using the starter template must either copy scripts manually or go without, both
of which are unacceptable as the toolset grows.

=== Requirements

1. **Zero manual copying** — a starter project must not require the developer
   to copy scripts from the monorepo.
2. **Version-pinned** — a project should be able to declare which version of
   the tools it uses, enabling controlled upgrades.
3. **Offline capable** — tools must work without an internet connection after
   initial setup.
4. **Same distribution model** — the mechanism should be consistent with how
   boundary libraries are already distributed (Clojars).
5. **Independently evolvable** — tools can be released on their own cadence,
   decoupled from the 20 domain libraries.

=== Considered approaches

[cols="1,2,2", options="header"]
|===
| Approach | Pros | Cons

| **Bundle scripts in starter template**
| Simple; zero infrastructure
| Scripts diverge from monorepo over time; updates require manual merges

| **Babashka URL task-import**
| Always latest version; no publishing step
| Requires internet at every run; no version pinning; fragile if URL changes

| **`boundary-tools` Clojars artifact** (chosen)
| Version-pinned; offline after first fetch; consistent with existing model;
  central release process
| Requires a publish step per release; slightly more setup than URL import
|===

== Decision

=== 1. New artifact — `org.boundary-app/boundary-tools`

All Babashka scripts are extracted from the monorepo into a dedicated,
independently publishable artifact published to Clojars:

----
boundary-tools/
├── bb.edn                          ; task definitions, references scripts/
├── deps.edn                        ; coordinates: org.boundary-app/boundary-tools
├── scripts/
│   ├── scaffold.clj                ; bb scaffold (moved from monorepo scripts/)
│   ├── ai.clj                      ; bb ai:* commands
│   ├── i18n.clj                    ; bb i18n:* commands
│   ├── admin.clj                   ; bb create-admin
│   ├── deploy.clj                  ; bb deploy
│   └── dev.clj                     ; bb check-links, smoke-check, install-hooks
└── AGENTS.md                       ; usage documentation
----

The monorepo's own `bb.edn` becomes a thin consumer of this artifact, identical
to how a starter project would use it, ensuring the monorepo and starters stay
in sync.

=== 2. Integration in starter projects

A `boundary-starter` project declares the tools artifact in `bb.edn` using
Babashka's Maven dependency support:

[source,clojure]
----
;; bb.edn in a boundary-starter project
{:deps {org.boundary-app/boundary-tools {:mvn/version "1.0.0"}}
 :tasks
 {:requires ([boundary.tools.scaffold :as scaffold]
             [boundary.tools.i18n     :as i18n]
             [boundary.tools.ai       :as ai]
             [boundary.tools.dev      :as dev])

  scaffold      {:doc "Interactive module scaffolding wizard"
                 :task (scaffold/run)}
  scaffold:ai   {:doc "NL scaffolding via AI"
                 :task (scaffold/run-ai)}

  ai:explain    {:doc "Explain a Clojure/Boundary stacktrace"
                 :task (ai/explain)}
  ai:gen-tests  {:doc "Generate a test namespace"
                 :task (ai/gen-tests)}
  ai:sql        {:doc "Generate HoneySQL from a description"
                 :task (ai/sql)}
  ai:docs       {:doc "Generate AGENTS.md for a module"
                 :task (ai/docs)}

  i18n:find     {:doc "Find translation key by string or key"
                 :task (i18n/find)}
  i18n:scan     {:doc "Scan for unexternalised UI strings"
                 :task (i18n/scan)}
  i18n:missing  {:doc "Report missing translations per locale"
                 :task (i18n/missing)}
  i18n:unused   {:doc "Report unused catalogue keys"
                 :task (i18n/unused)}

  create-admin  {:doc "Create first admin user"
                 :task (dev/create-admin)}
  check-links   {:doc "Validate AGENTS.md markdown links"
                 :task (dev/check-links)}
  smoke-check   {:doc "Verify deps.edn aliases and entrypoints"
                 :task (dev/smoke-check)}
  install-hooks {:doc "Configure git hooks path"
                 :task (dev/install-hooks)}}}
----

=== 3. Versioning and release cadence

`boundary-tools` follows the same versioning scheme as the domain libraries:
`MAJOR.MINOR.PATCH` with semantic versioning.  It is released independently
— a new scaffolding feature does not force a release of `boundary-user`.

The `bb deploy` task in the monorepo is extended to include
`boundary-tools` alongside the 20 domain libraries:

[source,bash]
----
bb deploy boundary-tools          ; deploy tools only
bb deploy --all                   ; deploy all 21 artifacts (20 libs + tools)
bb deploy --missing               ; deploy artifacts not yet on Clojars
----

=== 4. Monorepo alignment

The monorepo's own `bb.edn` is refactored to consume `boundary-tools` as a
local path dependency during development and the published artifact in CI:

[source,clojure]
----
;; monorepo bb.edn — development
{:deps {org.boundary-app/boundary-tools {:local/root "boundary-tools"}}}
----

This guarantees that the monorepo and starter projects run identical task
implementations.  Any script change in the monorepo goes through the normal
release process before reaching starter users.

=== 5. Upgrade path for existing starter projects

Upgrading the tools in a starter project is a one-line change in `bb.edn`:

[source,clojure]
----
;; bump version number
{:deps {org.boundary-app/boundary-tools {:mvn/version "1.1.0"}}}
----

A `bb smoke-check` after the bump verifies that all task entrypoints still
resolve correctly.

== Implementation Phases

=== Phase 1 — Extract and publish initial artifact
- [ ] Create `boundary-tools/` directory in the monorepo
- [ ] Move existing scripts (`scaffold.clj`, `ai.clj`, `deploy.clj`, etc.)
      into `boundary-tools/scripts/`
- [ ] Write `boundary-tools/bb.edn` with all task definitions
- [ ] Write `boundary-tools/deps.edn` with Maven coordinates
- [ ] Refactor monorepo `bb.edn` to consume `boundary-tools` as local path dep
- [ ] Verify all existing `bb *` commands still work in the monorepo
- [ ] Publish `org.boundary-app/boundary-tools 1.0.0` to Clojars
- [ ] Extend `bb deploy` to include `boundary-tools`

=== Phase 2 — Update boundary-starter template
- [ ] Replace ad-hoc `bb.edn` in `boundary-starter` with the artifact-based
      `bb.edn` shown in section 2
- [ ] Add `bb smoke-check` to the starter's CI pipeline
- [ ] Update `boundary-starter` README with `bb` command reference

=== Phase 3 — Add i18n tooling (follows ADR-013 Phase 1)
- [ ] Implement `bb i18n find`, `bb i18n scan`, `bb i18n missing`,
      `bb i18n unused` in `boundary-tools/scripts/i18n.clj`
- [ ] Release `boundary-tools 1.1.0`

=== Phase 4 — Documentation
- [ ] Write `boundary-tools/AGENTS.md` with full command reference
- [ ] Add upgrade guide to `boundary-starter` README
- [ ] Update root `AGENTS.md` to reference `boundary-tools` versioning

== Consequences

=== Positive

* Every boundary-starter project gets the full developer toolset from day one
  without any manual setup.
* Version pinning gives teams control over when they adopt new tooling —
  consistent with how they manage library upgrades.
* The monorepo and starter projects run identical script implementations,
  eliminating drift.
* New tools (e.g., future `bb i18n scan`) become available to all existing
  projects via a single version bump.
* The `bb deploy --all` command gives a single release command for the entire
  boundary ecosystem.

=== Negative

* Every new or changed tool requires a publish step before starter projects can
  use it — there is no "instant" distribution.
* Adds a 21st artifact to maintain and version alongside the 20 domain
  libraries.

=== Neutral

* Teams that do not use `boundary-starter` (e.g., the monorepo itself) see no
  change in workflow — the local path dependency is transparent.
* The `boundary-tools` artifact contains Babashka scripts only; it adds no
  JVM compile-time dependency to application code.

== Alternatives Considered

=== Bundle scripts in starter template

Copy `scripts/` and `bb.edn` directly into the starter at project-creation time.

*Rejected* because scripts diverge from the monorepo over time.  There is no
pull mechanism — teams either stay on stale scripts or merge manually.

=== Babashka URL task-import

Reference task definitions from a raw GitHub URL in `bb.edn`.

*Rejected* because it requires internet access on every run, offers no version
pinning, and breaks silently if the URL or branch changes.  It is also
incompatible with air-gapped or offline development environments.

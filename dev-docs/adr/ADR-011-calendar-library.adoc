= ADR-011: Calendar Library
:revdate: 2026-03-13
:revremark: boundary-calendar — persistence-agnostic calendar engine with ical4j, defevent, and pure Hiccup UI
:toc:
:toclevels: 3

== Status

**ACCEPTED** - Implemented and tested

== Context

The Boundary Framework Roadmap (Phase 2, Q3 2026) identified recurring calendar logic as the
fourth highest-value cross-cutting concern to extract into a reusable library.  Across
client projects, the same logic is written from scratch each time:

* Recurring event expansion with RFC 5545 RRULE
* Timezone-aware date arithmetic (DST transitions)
* Double-booking / conflict detection
* iCal feed generation (appointments, shifts, room bookings exported to Google Calendar, Outlook, iOS)
* iCal import from third-party sources
* Admin calendar views (month grid, week grid, mini navigator)

Estimated savings per project: **€3.2k – €4k** by eliminating this boilerplate.

=== Requirements

1. **Persistence-agnostic**: the library must never touch a database.
   Calendar events are owned by the application domain (appointments, bookings, shifts).
   The library provides compute on top of plain Clojure maps.

2. **RFC 5545 compliant**: export/import must round-trip correctly with Google Calendar,
   Outlook, and Apple Calendar, including RRULE, TZID, UID, and SUMMARY.

3. **DST-aware**: a 09:00 recurring meeting must remain at 09:00 local time after a
   clock change, not drift to 08:00 or 10:00.

4. **Pure core layer**: all business logic (recurrence, conflict, UI) must be in
   `core/` namespaces — no I/O, no side effects, no exceptions for valid inputs.

5. **Named event types**: follow the `defreport` pattern from `boundary-reports` so
   domain teams can register named event type schemas.

6. **Admin UI included**: pure Hiccup calendar view components composable into any
   Ring handler, consistent with `boundary-workflow/core/ui.clj`.

=== Alternatives Considered

==== Alternative A: Wrap Google Calendar / Outlook APIs

Rejected.  External API calls introduce network latency, authentication complexity,
rate limits, and hard vendor lock-in.  Most Boundary applications need an internal
calendar store with optional sync — not a read/write proxy to an external service.

==== Alternative B: clj-time / Joda-Time based RRULE expansion

Rejected.  `clj-time` is a legacy wrapper around Joda-Time.  The Boundary codebase
already uses `java.time` throughout (as of JDK 17+).  Writing RRULE parsing from
scratch against `java.time` would be hundreds of lines of error-prone edge-case code
(BYDAY ordinals, UNTIL parsing, EXDATE handling, COUNT).

==== Alternative C: Use ical4j 3.x

Rejected.  ical4j 3.x has a different API: `PropertyList.getProperty`, `ProdId` object
constructor, `Recur.getDates` takes `java.util.Date` (not `ZonedDateTime`), causing
DST-unsafe expansion.  Version 4.x was released in 2023, aligns with `java.time`, and
is the current stable release.

==== Alternative D: Dedicated CalDAV server (Radicale, Baikal)

Rejected.  CalDAV servers require separate infrastructure, docker containers, and
WebDAV access patterns that are incompatible with the in-process, zero-ops philosophy
of the Boundary Framework.

== Decision

We implement **`boundary-calendar`** as a pure-Clojure, persistence-agnostic calendar
library with the following key decisions:

=== Decision 1: ical4j 4.x as the sole RFC 5545 dependency

**Rationale**: ical4j is the de facto standard Java RFC 5545 library (>10 years, actively
maintained, used by Spring, Atlassian, and others).  Version 4.x introduces
`java.time`-native APIs making it the correct choice for JDK 17+ projects.

**Maven coordinate** (note: group ID changed in 4.x):

[source,clojure]
----
org.mnode.ical4j/ical4j {:mvn/version "4.0.3"}
----

WARNING: The artifact group changed from `net.fortuna.ical4j` (3.x) to `org.mnode.ical4j`
(4.x). The Java package names remain `net.fortuna.ical4j.*`.

**Key ical4j 4.x API facts** (relevant for maintainers):

[cols="50%,50%",options="header"]
|===
|ical4j 3.x |ical4j 4.x
|`new VEvent(Date, Date, String)` |`new VEvent(ZonedDateTime, ZonedDateTime, String)`
|`Calendar.withProdId(new ProdId(...))` |`Calendar.withProdId(String)`
|`getProperties()` returns `PropertyList` |`getProperties()` returns `java.util.List`
|`PropertyList.getProperty("DTSTART")` |`list.stream().filter(...)` — use `find-prop` helper
|`Recur.getDates(Date seed, ...)` |`Recur.getDates(ZonedDateTime seed, ...)` — DST-aware
|TZID resolved from `getParameters()` |TZID synthetic after parsing — parse from `(str property)`
|===

=== Decision 2: DST-aware occurrence expansion via ZonedDateTime seeds

**Problem**: when ical4j expands a `FREQ=WEEKLY;BYDAY=MO` rule seeded with a `java.util.Date`
(UTC), the expansion is UTC-anchored.  After a DST spring-forward (+1h offset change), a
09:00 local-time meeting appears to shift to 08:00 or 10:00 UTC, which then renders as
the wrong local time.

**Solution**: seed `Recur.getDates` with a `ZonedDateTime` in the event's `:timezone`
`ZoneId`.  ical4j 4.x honours the zone rules when iterating, so 09:00 local remains
09:00 local regardless of UTC offset changes.

[source,clojure]
----
(let [zone   (ZoneId/of (:timezone event))
      seed   (.atZone ev-start zone)          ; ZonedDateTime, not Instant
      zdts   (.getDates recur seed range-start range-end)]
  (->> zdts (map #(.toInstant ^ZonedDateTime %)) ...))
----

=== Decision 3: TZID extraction via regex, not getParameters

**Problem**: after `CalendarBuilder` parses an iCal string, ical4j resolves timezone
references to internal synthetic IDs like `"ical4j~b183c3b8-..."`.  Calling
`.getParameters()` on the parsed `DtStart` property returns these synthetic IDs, not
the original IANA names (e.g. `"Europe/Amsterdam"`).

**Solution**: parse the TZID parameter directly from the property's string
representation, which always contains the original IANA name:

[source,clojure]
----
;; Property text: "DTSTART;TZID=Europe/Amsterdam:20260310T090000"
(re-find #"TZID=([^;:\r\n]+)" (str dt-start-property))
;; => ["TZID=Europe/Amsterdam" "Europe/Amsterdam"]
----

=== Decision 4: Persistence-agnostic — library never stores data

The library operates on `EventData` maps provided by the caller.  The application
controls persistence (SQL, Datomic, EDN files, in-memory).  The library provides:

* Pure functions that compute on event maps
* iCal serialization/deserialization
* Conflict detection
* Occurrence expansion
* Hiccup UI components

This means the same library works for any persistence backend without modification.

=== Decision 5: `defevent` macro follows `defreport` pattern

The `defevent` macro is a direct parallel to `defreport` in `boundary-reports`.
It registers event type definitions in an in-process atom-backed registry.

[source,clojure]
----
;; Pattern from boundary-reports (libs/reports/src/boundary/reports/core/report.clj)
(defmacro defreport [sym definition-map]
  `(do (def ~sym ~definition-map)
       (register-report-type! ~sym)
       ~sym))

;; Calendar follows same shape
(defmacro defevent [sym definition-map]
  `(do (def ~sym ~definition-map)
       (register-event-type! ~sym)
       ~sym))
----

For clj-kondo: add `boundary.calendar.core.event/defevent clojure.core/def` to the
project's `.clj-kondo/config.edn` `:lint-as` map.

=== Decision 6: Shell service wraps adapter — callers never import ical4j

`boundary.calendar.shell.service` creates an `ICalAdapter` internally:

[source,clojure]
----
(defn export-ical
  ([events] (export-ical events {}))
  ([events opts]
   (let [adapter (get opts :adapter (ical-adapter/create-ical-adapter))]
     (ports/export-ical adapter events (dissoc opts :adapter)))))
----

Application code depends only on `boundary.calendar.shell.service`.  Swapping the
adapter (e.g. for a custom CalDAV-backed implementation) requires no changes to
callers.

=== Decision 7: Pure Hiccup UI components in `core/ui.clj`

Calendar view components live in the `core/` namespace (pure — no I/O) following the
pattern established by `boundary-workflow/core/ui.clj`.  All components accept plain
data and return Hiccup vectors.  No HTMX markup is added; composable into any Ring
handler or template.

Components: `event-badge`, `day-cell`, `month-view`, `week-view`, `mini-calendar`.

== Consequences

=== Benefits

1. **Eliminates boilerplate** in every project that needs calendar features (estimated
   saving: €3.2k–€4k/project).

2. **Correct DST handling** out of the box — the most common calendar bug is silently
   fixed by the library design.

3. **Zero infrastructure** — no external services, no Docker containers, no CalDAV
   server required.

4. **RFC 5545 compliant** — round-trips correctly with Google Calendar, Outlook, and
   Apple Calendar (verified by integration tests).

5. **Persistence-agnostic** — works with any application database or storage backend.

6. **Consistent with framework patterns** — `defevent` mirrors `defreport`; Hiccup UI
   mirrors `boundary-workflow/core/ui.clj`; test fixtures mirror `reports` tests.

7. **Composable conflict detection** — `find-conflicts` correctly handles pairs of
   recurring events by expanding both before comparison.

=== Drawbacks

1. **ical4j 4.x API is not well-documented**.  Migration from 3.x required iterative
   debugging to discover the API changes documented in Decision 1 above.  Future
   maintainers must consult AGENTS.md before touching `ical.clj`.

2. **TZID extraction is a workaround**.  The regex approach in Decision 3 is fragile
   if ical4j changes its `property.toString()` format.  A more robust solution would
   use the ical4j `TimeZone` registry API, but the current approach is correct for
   all tested inputs.

3. **O(n²) conflict detection**.  `find-conflicts` is quadratic in the number of
   events.  For typical calendar windows (< 200 events/week) this is unnoticeable.
   Applications with thousands of concurrent events should implement indexing at the
   persistence layer before calling the library.

4. **100-year occurrence window** in `next-occurrence*`.  The 36524-day search window
   is generous but not infinite — events scheduled beyond the year 2126 will not be
   returned by `next-occurrence*`.

=== Risks and Mitigations

==== Risk: ical4j synthetic timezone IDs break on future versions

**Mitigation**: `get-tzid` tests cover the `TZID=Europe/Amsterdam` regex pattern.
If ical4j changes `property.toString()` format, the test `round-trip-non-recurring-test`
will catch it immediately.

==== Risk: BYDAY ordinals not supported in conflict expansion

**Mitigation**: `FREQ=MONTHLY;BYDAY=1MO` (first Monday of month) is supported via
`Recur.getDates()`.  Documented in AGENTS.md as a supported pattern.  Complex
`EXDATE`/`EXRULE` exclusions are not explicitly handled — they are silently passed
through to ical4j which respects them during expansion.

==== Risk: Registry pollution between tests

**Mitigation**: `clear-registry!` function provided.  All tests that use `defevent`
must call it in an `:each` fixture.  Documented prominently in AGENTS.md.

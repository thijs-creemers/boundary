= ADR-025: Zero-Friction Onboarding
:status: Proposed
:date: 2026-04-18
:deciders: boundary-team

== Status

Proposed

== Context

=== Background

Boundary provides `bb setup` for interactive configuration and `bb scaffold`
for module generation. However, a new developer must discover and chain these
commands manually. There is no single command that takes them from a fresh
clone to a running application.

=== Problem

* First-run experience requires multiple manual steps: setup, scaffold, migrate,
  start REPL, navigate to URL.
* Environment prerequisites (Java version, Clojure CLI, ports) are not checked
  upfront -- developers discover missing tools through cryptic errors.
* No IDE-specific configuration ships with the project -- developers must
  manually configure nREPL connections.
* The existing `bb doctor` validates config but not system prerequisites.

== Decision

=== `bb quickstart` command

A single command that orchestrates the full onboarding flow:

1. Run `bb doctor:env` to check prerequisites
2. Run `bb setup` interactively (or `--preset minimal|standard|sqlite` for non-interactive)
3. Validate configuration via `bb doctor`
4. Scaffold a sample module ("tasks" with title and done fields)
5. Run database migrations
6. Verify project structure via `bb smoke-check`

Presets map to concrete setup flags: `minimal` = H2 in-memory, `standard` =
PostgreSQL, `sqlite` = SQLite file-based.

NOTE: Quickstart does not auto-start the system or open the browser. Starting a
REPL is a long-running foreground process that belongs in the developer's
terminal. The success output tells the user exactly what to do next.

=== `bb doctor:env` environment checker

Complements the existing `bb doctor` (config validation):

* `bb doctor` = config validation (existing, unchanged)
* `bb doctor:env` = system prerequisites (new)
* `bb doctor --all` = both combined

Checks: Java version (minimum 17), Clojure CLI, Babashka, Node.js, port
availability (3000, 7888, 9999), database drivers, clj-kondo, AI providers
(Ollama, MLX, OpenAI-compatible endpoints).

=== IDE bootstrapping

Auto-generated by `bb setup` based on detected editor:

* **VS Code**: `.vscode/extensions.json` + `.vscode/settings.json`
* **IntelliJ/Cursive**: `.idea/runConfigurations/`
* **Emacs/CIDER**: `.dir-locals.el`
* **Neovim/Conjure**: `.conjure/` with nREPL connection config

=== Local example app

`examples/quickstart/` -- a minimal app (1 module, SQLite, no auth) that lives
in-repo as part of the onboarding flow. Not a showcase app; kept minimal to
avoid monorepo bloat.

=== AI provider flexibility

Support multiple local and cloud AI providers:

* Ollama (default), MLX/oMLX (macOS native), Anthropic, OpenAI,
  any OpenAI-compatible endpoint (LM Studio, vLLM, etc.)
* Auto-detection in `bb doctor:env`

== Consequences

=== Positive

* *Minutes to first endpoint* -- one command from clone to running app.
* *Fewer "works on my machine" issues* -- environment prereqs checked upfront.
* *Editor-ready* -- IDE config eliminates manual nREPL setup.
* *Local-first AI* -- MLX and Ollama support means no data leaves the machine.

=== Negative / risks

* *`bb quickstart` scope creep* -- must stay simple and fast. If it takes
  more than 2 minutes, it defeats the purpose.
* *IDE config maintenance* -- editor configurations change across versions.

=== Mitigation

* `bb quickstart` delegates to existing commands (`setup`, `scaffold`,
  `migrate`) rather than reimplementing them.
* IDE configs are generated, not hand-maintained -- templates can be updated
  centrally.

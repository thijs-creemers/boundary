== Overview

The user module (`libs/user`) is Boundary's reference implementation of the FC/IS pattern with multi-tenancy support. It handles authentication, session management, user profiles, MFA, and audit logging.

=== Current Module Structure

[source]
----
libs/user/src/boundary/user/
├── schema.clj               # Malli schemas: AuthUser, TenantUser, User, UserSession, UserAuditLog
├── ports.clj                # Port protocols: IUserRepository, IUserSessionRepository, etc.
├── core/
│   ├── user.clj             # Pure user entity logic
│   ├── session.clj          # Pure session logic
│   ├── authentication.clj   # Pure authentication logic (password, lockout, MFA flow)
│   ├── mfa.clj              # Pure MFA logic (TOTP, backup codes)
│   ├── audit.clj            # Pure audit entry constructors
│   ├── validation.clj       # Reusable validation helpers
│   ├── ui.clj               # Hiccup UI components (user listing, forms)
│   └── profile_ui.clj       # Hiccup profile UI components
└── shell/
    ├── service.clj           # UserService — orchestrates core + persistence
    ├── persistence.clj       # IUserRepository + IUserSessionRepository (DB adapter)
    ├── auth.clj              # AuthenticationService implementation
    ├── auth_persistence.clj  # AuthUserRepository — reads/writes public.auth_users
    ├── mfa.clj               # MfaService implementation
    ├── http.clj              # REST API + Web UI route definitions
    ├── web_handlers.clj      # HTMX web UI handlers
    ├── http_interceptors.clj # HTTP-layer interceptors (auth, audit context)
    ├── interceptors.clj      # Service-layer interceptors (session list pipeline)
    ├── middleware.clj        # Ring middleware (request context binding)
    ├── cli.clj               # CLI command definitions
    ├── cli_entry.clj         # CLI entry point
    └── module_wiring.clj     # Integrant component definitions
----

'''

== Multi-Tenancy Architecture

=== Split-Table Design (ADR-004)

The user module implements a *split-table architecture* to support schema-per-tenant multi-tenancy cleanly:

[cols="1,2,3"]
|===
|Table |Schema Location |Contents

|`auth_users`
|`public` (shared, global)
|Email, password hash, MFA state, failed login count, lockout

|`users`
|`tenant_<slug>` (per-tenant, isolated)
|Name, role, avatar URL, preferences (theme, timezone, notifications), login count

|`user_sessions`
|Depends on deployment
|Session tokens, expiry, IP, user agent

|`user_audit_log`
|Depends on deployment
|Full audit trail: action, actor, target, changes, result
|===

A single user ID exists in *one row* of `public.auth_users` and *one row per tenant* in `tenant_<slug>.users`. The same person logging into two different tenants gets different name, role, and preference settings.

=== The Three Schema Entities

[source,clojure]
----
;; Global — lives in public.auth_users
(def AuthUser
  [:map
   [:id :uuid]
   [:email ...]
   [:password-hash :string]
   [:active :boolean]
   [:mfa-enabled :boolean]
   [:mfa-secret [:maybe :string]]
   [:mfa-backup-codes [:maybe [:vector :string]]]
   [:failed-login-count :int]
   [:lockout-until [:maybe inst?]]
   ;; + audit timestamps
   ])

;; Per-tenant — lives in tenant_<slug>.users
(def TenantUser
  [:map
   [:id :uuid]
   [:tenant-id [:maybe :uuid]]
   [:name :string]
   [:role [:enum :admin :user :viewer]]
   [:avatar-url :string]
   [:login-count :int]
   ;; + flattened preferences (notifications, theme, language, timezone, date/time format)
   ;; + audit timestamps
   ])

;; Merged domain entity — used throughout business logic
(def User
  ;; AuthUser fields + TenantUser fields merged into one map
  ;; The persistence layer is responsible for JOIN and field routing on writes
  )
----

=== Non-Tenant Mode

When the `boundary-tenant` module is *not* active, the framework operates in single-tenant mode:

* The `users` table lives in the default public schema rather than a `tenant_<slug>` schema
* `:tenant-id` fields are `nil` or omitted throughout
* All other module behaviour (auth, sessions, MFA, audit) is identical

The persistence layer handles both modes transparently via the database context (`ctx`) passed at initialisation — no conditional branching is needed in service or core code.

== Schemas (`boundary.user.schema`)

=== Domain Entities

[cols="1,3"]
|===
|Schema |Purpose

|`AuthUser`
|Persisted to `public.auth_users`. Holds global auth credentials. Never exposed directly via API.

|`TenantUser`
|Persisted to `tenant_<slug>.users`. Holds tenant-specific profile and preferences.

|`User`
|Merged domain entity used everywhere in business logic. Combines fields from both tables.

|`UserSession`
|One session per login. Holds token, expiry, IP, user agent, revocation timestamp.

|`UserAuditLog`
|Immutable record of every user-affecting action (create, update, delete, login, logout, role-change, bulk-action).
|===

=== Request / Response Schemas

[cols="1,2"]
|===
|Schema |Description

|`CreateUserRequest`
|New user: email, name, password, role, optional preferences. `:closed true` — rejects unknown keys.

|`UpdateUserRequest`
|Partial update: all fields optional. Routes auth fields to `auth_users`, profile fields to `users`.

|`LoginRequest`
|Email + password + optional MFA code, remember flag, IP, user-agent.

|`MFASetupRequest` / `MFAEnableRequest` / `MFAVerifyRequest` / `MFADisableRequest`
|MFA lifecycle schemas.

|`UserResponse`
|Public user data in camelCase. Sensitive fields (`password-hash`, `mfa-secret`, `mfa-backup-codes`) are always stripped.

|`LoginResponse`
|Auth result. May carry `:requiresMfa true` before TOTP verification, then `:sessionToken` on success.

|`MFASetupResponse`
|TOTP secret, QR code data URL, backup codes list.
|===

=== Transformation Pipeline

[source]
----
HTTP/CLI Input (camelCase)
    │
    ▼  user-request-transformer
    │  • camelCase → kebab-case
    │  • string enums → keywords
    │  • string booleans → booleans
    ▼
Domain Entity (kebab-case)
    │
    │  Business logic in core.*
    ▼
Domain Entity (kebab-case)
    │
    ▼  user-response-transformer
    │  • kebab-case → camelCase
    │  • inst? → ISO 8601 string
    │  • UUID → string
    │  • keyword → string enum
    │  • strip sensitive fields
    │  • remove nil values
    ▼
API Response (camelCase)
----

== Ports (`boundary.user.ports`)

[cols="1,3"]
|===
|Protocol |Responsibility

|`IUserRepository`
|CRUD on the merged `User` entity. Writes are split across `auth_users` and `users` internally.

|`IUserSessionRepository`
|Session lifecycle: create, validate by token, invalidate, invalidate all, list by user.

|`IAuthUserRepository`
|Direct access to `public.auth_users` for authentication: find by email, update password, update MFA state, update lockout.

|`IAuditRepository`
|Write and query `user_audit_log`. Supports filtering by actor, target, action type, date range.

|`IEmailService`
|Transactional email (welcome, password reset, notifications).

|`INotificationService`
|Multi-channel notification delivery (email, push, SMS, in-app).
|===

== Shell Components

=== Service Layer (`service.clj`)

The `UserService` record coordinates the full user lifecycle:

[source,clojure]
----
(defrecord UserService
  [user-repository
   session-repository
   audit-repository
   validation-config
   auth-service       ; delegates auth to AuthenticationService
   cache])            ; optional Redis cache for session/user lookups
----

Key responsibilities:

* User CRUD — delegates persistence to `IUserRepository`, business rules to `core.user`
* Session management — delegates to `IUserSessionRepository` + `core.session`
* Audit recording — every mutating operation produces an `UserAuditLog` entry via `core.audit`
* Cache integration — session tokens and user lookups optionally cached in Redis

The `*audit-context*` dynamic var is bound per-request by HTTP/CLI interceptors and carries `:actor-id`, `:actor-email`, `:ip-address`, and `:user-agent` for automatic audit trail population.

=== Authentication Service (`auth.clj`)

Separate from `UserService` to keep concerns clear:

[source,clojure]
----
(defrecord AuthenticationService
  [user-repository
   session-repository
   mfa-service
   auth-config])
----

Handles:

* Password verification via `core.authentication`
* Account lockout (N failed attempts → lockout until timestamp)
* MFA flow: if enabled, login returns `:requiresMfa true` after password check; second call with `:mfa-code` completes auth
* Session creation on successful full authentication

=== Auth Persistence (`auth_persistence.clj`)

Dedicated repository for `public.auth_users`. Keeps auth-specific queries (password lookup, MFA state, lockout) separate from the profile queries in `persistence.clj`.

=== MFA Service (`mfa.clj`)

Implements TOTP-based MFA:

* Setup: generates Base32 secret + QR code URL + backup codes
* Enable: verifies TOTP code before activating
* Verify: accepts 6-digit TOTP code or a one-time backup code
* Disable: requires current password (and optionally current TOTP code)

=== Persistence Layer (`persistence.clj`)

Implements `IUserRepository` and `IUserSessionRepository` against the database context. Key design points:

* **Split writes**: `create-user` inserts into `auth_users` first (establishes the UUID), then into the tenant `users` table
* **Merged reads**: queries JOIN `auth_users a` with `users u` on `a.id = u.id` to return a flat `User` map
* **Update routing**: `update-user` inspects which fields belong to which table and issues separate UPDATE statements
* **Tenant-aware queries**: tenant context is carried in the `ctx` (database context), not as an explicit parameter on every call
* **Schema initialisation**: `initialize-user-schema!` creates all four tables from Malli schema definitions via `db-schema/initialize-tables-from-schemas!`

=== HTTP Layer (`http.clj`)

== Observability

All observability is handled automatically by interceptors — there are no manual `log/info` or metrics calls in service or core functions for routine operations.

* **HTTP layer**: `http_interceptors.clj` binds audit context, handles auth checks
* **Service layer**: `boundary.platform.shell.service-interceptors` wraps service methods for entry/exit logging, timing histograms, and error reporting
* **Persistence layer**: `boundary.platform.shell.persistence-interceptors` wraps DB operations for query logging and timing
* **Audit trail**: `*audit-context*` dynamic var carries actor identity from the request into the service layer without threading it through every function signature

== Data Flow: Login with MFA

[source]
----
POST /api/auth/login  {email, password}
    │
    ▼  HTTP interceptor: bind audit context (ip, user-agent)
    │
    ▼  AuthenticationService.authenticate
    │   ├── auth_persistence: find-by-email → AuthUser row
    │   ├── core.authentication: verify-password (bcrypt)
    │   ├── core.authentication: check-lockout
    │   └── core.authentication: mfa-required? → true
    │
    ▼  HTTP response: {requiresMfa: true}

POST /api/auth/login  {email, password, mfaCode}
    │
    ▼  AuthenticationService.authenticate
    │   ├── mfa-service: verify-totp-code
    │   ├── persistence: create-session → UserSession
    │   ├── service: write audit log entry (action: :login, result: :success)
    │   └── cache: store session token (if cache configured)
    │
    ▼  HTTP response: {sessionToken, user: {...}}
----

'''

== Data Flow: Create User (Multi-Tenant)

[source]
----
POST /api/users  {email, name, password, role}
    │
    ▼  Validation: CreateUserRequest schema (camelCase → kebab-case)
    │
    ▼  UserService.create-user
    │   ├── core.user: build-new-user (generate UUID, hash password, defaults)
    │   ├── auth_persistence: insert into public.auth_users (email, password-hash, active)
    │   ├── persistence: insert into tenant_<slug>.users (name, role, preferences)
    │   ├── service: write audit log entry (action: :create, result: :success)
    │   └── (optional) email-service: send welcome email
    │
    ▼  Response: UserResponse (sensitive fields stripped, camelCase)
----

In single-tenant mode the second insert targets `public.users` instead of `tenant_<slug>.users`. Everything else is identical.

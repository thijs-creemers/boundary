=== Het "Vette model": Een Liefdesverhaal

Het begint altijd zo onschuldig.

[source,python]
----
# models.py, Jaar 1 — schoon, verstandig
class User(AbstractBaseUser):
    email    = models.EmailField(unique=True)
    password = models.CharField(max_length=128)
----

[source,python]
----
# models.py, Jaar 3 — "het leek een goed idee"
class User(AbstractBaseUser):

    def register(self, email, password, plan="free"):
        if not self._is_allowed_domain(email):      # bedrijfsregel
            raise ValidationError("Domein niet toegestaan")

        self.password = make_password(password)      # hashing (I/O)

        if User.objects.filter(email=email).exists(): # databasequery
            raise ValidationError("Al geregistreerd")

        send_mail("Welkom!", "...", FROM, [email])   # e-mail (netwerk-I/O)
        stripe.Customer.create(email=email)           # facturering (externe API)
        self.save()                                   # opslaan (I/O)
----

[.notes]
--
* "Jaar 1 is altijd mooi. Jaar 3 is wanneer de sollicitatiegesprekken beginnen."
* "Niemand *besloot* Stripe in het model te stoppen. Het is er gewoon... terecht gekomen. Zwaartekracht."
* Vraag het publiek: "Wie heeft een model dat meer dan 3 van deze dingen doet? Houd je hand op. ... Ja."

=== Rails Callbacks: Magie met Gevolgen

[source,ruby]
----
# app/models/user.rb
class User < ApplicationRecord
  before_create :hash_password
  after_create  :send_welcome_email   # vuurt... soms. in een transactie. misschien.
  after_create  :charge_stripe         # want wat kan er misgaan
  after_commit  :invalidate_cache      # op welke verbinding? goede vraag.

  def register!(email, password, plan: :free)
    raise "Domein niet toegestaan" unless allowed_domain?(email)
    raise "Al in gebruik"          if User.exists?(email: email)
    self.email = email
    self.plan  = plan
    save!
  end
end
----

[quote]
____
`after_create` — perfect voor neveneffecten die je bijna altijd wilt, behalve in tests, en mogelijk in productie op dinsdagen.
____

[.notes]
--
* "Rails is elegant en mooi. ActiveRecord is een meesterwerk... voor de eerste 18 maanden van een project."
* "Callbacks zijn de plek waar logica onzichtbaar wordt. Je roept `save!` aan en je weet niet meer wat er gebeurt."
* "VCR-cassettes, `ActionMailer::Base.deliveries`, let(:stripe_mock) — de testsuite begint op een toneelproductie te lijken."

=== Spring Boot — Hetzelfde Verhaal, Enterprise-editie

Spring Boot heeft dependency injection en een servicelaag. Probleem opgelost, toch? +
_Niet helemaal._

[source,java]
----
// UserService.java — een volstrekt normale Spring-service
@Service
@Transactional
public class UserService {

    @Autowired private UserRepository  userRepository;   // JPA — database
    @Autowired private PasswordEncoder passwordEncoder;  // bcrypt — CPU-I/O
    @Autowired private JavaMailSender  mailSender;       // SMTP — netwerk-I/O
    @Autowired private StripeClient    stripeClient;     // REST — externe API

    public User register(RegisterRequest req) {
        // Bedrijfsregel + databaseaanroep in één adem
        if (userRepository.existsByEmail(req.email()))
            throw new EmailAlreadyExistsException();

        if (!isAllowedDomain(req.email()))               // pure bedrijfsregel
            throw new DomainNotAllowedException();       // zit vast in dezelfde klasse

        String hash = passwordEncoder.encode(req.password()); // I/O
        User user   = userRepository.save(                    // I/O
                          new User(req.email(), hash));

        sendWelcomeEmail(user);                          // I/O
        stripeClient.createCustomer(user.getEmail());    // I/O

        return user;
    }
}
----

DI is prima. Maar `UserService` *kent* nog steeds bedrijfsregels ÉN *doet* I/O — in dezelfde klasse.

[.notes]
--
* "Spring-devs zeggen: 'maar we hebben interfaces en DI!' Klopt — en dat is al beter dan ActiveRecord. Maar de *service* mengt nog steeds logica en I/O."
* "De `isAllowedDomain`-regel woont in dezelfde klasse als `stripeClient.createCustomer`. Ze hebben niets met elkaar te maken."
* "`@Transactional` is een verborgen neveneffect. Je roept `register()` aan en een transactie wordt onzichtbaar geopend."
* "De service *orkestreert* én *beslist* in één adem. FC/IS zegt: dat hoort thuis in aparte namespaces ."
--

=== De belasting op tests — Alle Drie de Stacks

Om *één* bedrijfsregel te testen — "is dit e-maildomein toegestaan?" — heb je nodig:

.Django
[source,python]
----
@patch("myapp.models.stripe.Customer.create")   # niet echt afschrijven
@patch("myapp.models.send_mail")                # niet echt mailen
@patch("myapp.models.make_password", return_value="hashed")
@pytest.mark.django_db
def test_domain_not_allowed(mock_hash, mock_mail, mock_stripe):
    user = User()
    with pytest.raises(ValidationError):
        user.register("alice@concurrent.com", "s3cr3t")
    mock_stripe.assert_not_called()
    mock_mail.assert_not_called()
----

.Spring Boot
[source,java]
----
@ExtendWith(MockitoExtension.class)    // in ieder geval geen volledige Spring-context...
class UserServiceTest {

    @Mock UserRepository  userRepository;  // vier mocks
    @Mock PasswordEncoder passwordEncoder;
    @Mock JavaMailSender  mailSender;
    @Mock StripeClient    stripeClient;
    @InjectMocks UserService userService;

    @Test void rejectsDomainNotAllowed() {
        when(passwordEncoder.encode(any())).thenReturn("hashed");
        when(userRepository.existsByEmail(any())).thenReturn(false);

        assertThrows(DomainNotAllowedException.class, () ->
            userService.register(new RegisterRequest("alice@concurrent.com", "s3cr3t")));

        verify(stripeClient, never()).createCustomer(any()); // paranoia
        verify(mailSender,   never()).send(any());           // nog meer paranoia
    }
}
----

Vier mocks in beide gevallen — alleen maar om een domeincheck te testen die niets met Stripe of e-mail te maken heeft.

[.notes]
--
* "De Spring-versie heeft geen `@SpringBootTest` nodig als je `@ExtendWith(MockitoExtension.class)` gebruikt — terecht. Maar vier Mockito-mocks voor een string-check is nog steeds de verkeerde abstractie."
* "Elke keer dat de service een nieuwe dependency krijgt, moet elke test in deze klasse worden bijgewerkt."
* "De mocks testen *of je eraan gedacht hebt Stripe niet aan te roepen* — niet de bedrijfsregel zelf."
* "De architectuur heeft deze testlast gecreëerd. De architectuur kan het ook oplossen."

=== Weten vs. Doen

Kijk nog eens naar die `register()`-methode. Ze doet twee volledig verschillende *soorten* dingen:

[cols="1a,1a", options="header"]
|===
| 🧠 *Weten* (pure logica) | 🔌 *Doen* (neveneffecten)

| Is dit e-maildomein toegestaan?
| Zoek bestaande gebruikers in de database

| Is het wachtwoord sterk genoeg?
| Hash het wachtwoord (bcrypt — langzaam by design)

| Moet dit account worden vergrendeld?
| Schrijf de vergrendeling naar de database

| Hoe lang moet deze sessie duren?
| Genereer een cryptografisch token

| Is deze inlogpoging verdacht?
| Stuur een beveiligingswaarschuwing per e-mail
|===

[.callout]
****
Elk item in de linkerkolom kan worden beantwoord met *alleen data*. +
Geen database. Geen netwerk. Geen tijd. Geen willekeur. +
Gewoon functies.
****

[.notes]
--
* "Dit is het aha-moment dat ik je wil geven: de linkerkolom is *wiskunde*. Het is gewoon logica."
* "Je kunt elk item in de linkerkolom testen door wat data-maps mee te geven en het resultaat te controleren."
* "De rechterkolom is waar de wereld erbij komt. Daar worden dingen duur en traag."
* "FC/IS zegt: laten we deze twee kolommen niet door elkaar mengen in dezelfde functie."

// ─────────────────────────────────────────────────────────────────────────────
// SLIDE — TWEE REGELS, DAT IS ALLES
// ─────────────────────────────────────────────────────────────────────────────
=== Twee Regels, Dat Is Alles

[cols="1a,1a", options="header"]
|===
| ✅ Functional Core | ✅ Imperative Shell

| Alleen pure functies +
Geen I/O, geen logging, geen uitzonderingen voor flow control +
Zelfde input → zelfde output, altijd, voor altijd +
Afhankelijk van alleen data en andere pure functies

| Alle neveneffecten leven hier — en *alleen* hier +
Leest DB, roept API's aan, logt, hasht, verstuurt e-mails +
Levert alle "buitenwereld"-inputs aan de core +
Componeert core-aanroepen tot workflows
|===

IMPORTANT: Shell → Core is toegestaan. Core → Shell is *nooit* toegestaan.

Dat is het hele patroon. +
Serieus. Dat is het. De rest is implementatie.

[.notes]
--
* "Ik weet dat het te simpel klinkt. Maar de implicaties zijn enorm."
* "In Boundary wordt deze regel afgedwongen door de linter. Een compile-time check. Als een `core/`-namespace iets vereist uit `shell/`, mislukt de build."
* "In Python en Ruby dwing je het af met code reviews en discipline. Minder betrouwbaar, maar nog steeds effectief."
--

=== Wat Hexagonaal Toevoegt

FC/IS vertelt je *waar* logica leeft. Hexagonale Architectuur (Alistair Cockburn, 2005) vertelt je *hoe de shell verbinding maakt met de buitenwereld*.

Het kernidee: alle verbindingen met externe systemen verlopen via **ports**.

* Een **port** is een *interface* (een contract, een protocol, een ABC — hoe je taal het ook noemt)
* Een **adapter** is een *concrete implementatie* van die port
* Het domein kent alleen de port. Het importeert nooit een adapter.

[mermaid,format=svg,width=100%]
....
flowchart LR
    HTTP["🌐 HTTP"]
    CLI["💻 CLI"]
    TESTS["🧪 Tests"]

    SVC(["IUserService\n— input port —"])

    CORE["🧠 Bedrijfslogica\npure functies"]

    REPO(["IUserRepository\n— output port —"])
    MAIL(["IEmailService\n— output port —"])

    PG[("PostgreSQL")]
    H2[("H2")]
    SMTP["📧 SMTP"]
    SES["☁️ AWS SES"]

    HTTP & CLI & TESTS --> SVC --> CORE
    CORE --> REPO & MAIL
    REPO --> PG & H2
    MAIL --> SMTP & SES

[.notes]
--
* "De testsuite is een driving adapter. Hij stuurt het domein aan via dezelfde interface als HTTP."
* "Dat is het kernidee: tests en productiecode gebruiken *dezelfde deur*."
* "En omdat de output ports gewoon interfaces zijn, kun je H2 inpluggen voor tests en PostgreSQL in productie. Geen codewijzigingen."

=== Een Port in Boundary

[source,clojure]
----
;; libs/user/src/boundary/user/ports.clj

(defprotocol IUserRepository
  "Output port — de kijk van het domein op 'iets dat gebruikers opslaat'.
   Het weet niet of dat PostgreSQL, H2 of een map in geheugen is."

  (find-user-by-id    [this user-id])
  (find-user-by-email [this email])
  (create-user        [this user-data])
  (update-user        [this user-data])
  (soft-delete-user   [this user-id])
  (find-users         [this options]))
----

Vergelijk: in Django *is* je model de ORM. In Rails *erf* je van `ActiveRecord::Base`. +
Beide koppelen je bedrijfslogica aan een specifieke database-library op klasseniveau.

Een port zegt: "Ik heb *iets* nodig dat gebruikers kan vinden en opslaan. Ik geef niet om wat het is."

[.notes]
--
* "In Python gebruik je `typing.Protocol` of `abc.ABC`. In Ruby, duck typing + documentatie — minder expliciet maar werkbaar."
* "Het cruciale verschil: met een port kun je de implementatie vervangen zonder bedrijfslogica aan te raken."
* "Met Django ORM kun je het mocken. Dat is anders dan het vervangen."

=== De 13 Libraries

Boundary is geen monoliet. Het zijn 13 onafhankelijk publiceerbare libraries, elk gericht op *één ding*:

[cols="1,1,3", options="header"]
|===
| Categorie | Library | Wat het doet

.3+| *Fundament*
| `core`
| Validatie, hoofdletterconversie, interceptor-pipeline, feature flags

| `platform`
| HTTP-routing (Reitit), DB-adapters (PostgreSQL / MySQL / SQLite / H2), migraties, multi-DB-ondersteuning

| `observability`
| Request-logging, gestructureerde metrics, distributed tracing-interceptors

.3+| *Domein*
| `user`
| Authenticatie, sessies, MFA, wachtwoordbeleid, audittrail

| `admin`
| HTMX-aangedreven back-office UI, entiteitsconfiguratie, tabel/formulier-generatie

| `tenant`
| Multi-tenancy (schema-per-tenant), tenant-resolutie-middleware

.5+| *Infrastructuur*
| `cache`
| Redis + in-memory adapters, TTL-beheer, atomaire operaties

| `jobs`
| Achtergrondtaakverwerking, retry-logica, dead-letter queues

| `email`
| SMTP-verzending, async-wachtrijmodus, in-memory adapter voor tests

| `storage`
| Lokaal bestandssysteem + AWS S3-adapters, afbeeldingsverkleining, presigned URLs

| `realtime`
| WebSocket-verbindingen, pub/sub-messaging, SSE-ondersteuning

.1+| *Tooling*
| `scaffolder`
| Genereert een nieuw FC/IS-module met de juiste directorystructuur in één commando

.1+| *Externe services*
| `external`
| Adapters voor SMTP, IMAP, Stripe-betalingen & webhooks, en Twilio SMS/WhatsApp — elk achter zijn eigen port
|===

[.notes]
--
* "Elke library heeft zijn eigen `build.clj`, zijn eigen tests, zijn eigen versie. Je kunt alleen `boundary/user` naar Clojars publiceren."
* "Je hoeft niet alle 13 te gebruiken. Pak wat je nodig hebt. Het patroon houdt ze consistent."
* "De scaffolder genereert een nieuwe library met de juiste FC/IS-structuur in één commando."

=== De Database Is Gewoon een Adapter

Boundary wordt geleverd met adapters voor vier databases — die allemaal dezelfde port implementeren:

[source,clojure]
----
;; Vier adapters. Één port. Nul wijzigingen in bedrijfslogica als je wisselt.

;; PostgreSQL / H2 / MySQL / SQLite adapter — zelfde code, andere driver
(defrecord SQLUserRepository [ctx]           ; ctx bevat db-verbinding + dialect
  IUserRepository
  (find-user-by-email [_ email]
    (db/execute-one! ctx                     ; next.jdbc onder de motorkap
      {:select [:id :email :password-hash :active :role]
       :from   [:auth-users]
       :where  [:= :email email]})))         ; HoneySQL map — geparametriseerd, geen string-concat

;; In-memory adapter — ultrasnelle unit tests, nul I/O
(defrecord InMemoryUserRepository [store]
  IUserRepository
  (find-user-by-email [_ email]
    (first (filter #(= email (:email %)) (vals @store)))))
----

De HoneySQL-map compileert naar een geparametriseerde prepared statement — `[:= :email email]` doet nooit string-concatenatie.

[.notes]
--
* "In Django *is* je model de ORM. Als je zonder database wilt testen, mock je de ORM. Dat is shimmen."
* "In Boundary is de database een stekker. Je trekt de ene stekker eruit en steekt een andere erin. Geen shimmen nodig."
* "De HoneySQL-map vermijdt raw SQL-strings volledig — geen SQL-injection-aanvalsoppervlak, en dezelfde map werkt voor alle vier databasedialecten."
* "De in-memory adapter maakt unit tests voor shell-services triviaal — geen Docker, geen migraties."
--

=== Gebruikersregistratie: Boundary Shell (Orkestratie)

[source,clojure]

;; libs/user/src/boundary/user/shell/service.clj — vereenvoudigd
;; De shell is een script: ophalen → beslissen → handelen → herhalen.

(defn register-user! [repositories user-data]
  (let [{:keys [user-repo audit-repo]} repositories]

    ;; Stap 1: CORE — pure validatie, geen I/O
    (let [result (user-core/validate-user-creation-request user-data config)]
      (when-not (:valid? result)
        (throw (ex-info "Ongeldige data" {:errors (:errors result)}))))

    ;; Stap 2: SHELL haalt op, CORE beslist — logica blijft puur
    (let [existing (user-repo/find-user-by-email user-repo (:email user-data))
          decision (user-core/check-duplicate-user-decision user-data existing)]
      (when (= :reject (:decision decision))
        (throw (ex-info "Gebruiker bestaat al" {}))))

    ;; Stap 3: SHELL doet de dure I/O, geeft resultaten door aan CORE
    (let [hash     (auth-shell/hash-password (:password user-data))  ;; bcrypt hier
          now      (current-timestamp)    ;; shell bezit de klok
          id       (generate-user-id)     ;; shell bezit willekeurigheid
          prepared (user-core/prepare-user-for-creation
                     (assoc user-data :password-hash hash) now id)

          ;; Stap 4: SHELL slaat op
          created  (user-repo/create-user user-repo prepared)]

      (dissoc created :password-hash))))  ;; lek de hash nooit
----

[.notes]
--
* "De shell is bijna saai. Het is een recept: haal ingrediënten op, roep een core-functie aan, sla het resultaat op."
* "Alle interessante logica zit in de core. De shell orkestreert alleen."
* "Let op: `current-timestamp` en `generate-user-id` zijn gedefinieerd in de shell. De core krijgt het *resultaat*, niet de functie."

(defn analyze-login-risk
  "Score the risk of a login attempt. No database, no HTTP, no Redis — just data."
  [user login-context recent-sessions now]
  (let [risk-factors
        (cond-> []
          ;; Inloggen vanaf een nieuw IP-adres? Verdacht.
          (not-any? #(= (:ip-address login-context) (:ip-address %)) recent-sessions)
          (conj {:factor :new-ip-address :weight 30})

          ;; Admin-account? Hogere drempel.
          (= :admin (:role user))
          (conj {:factor :admin-user :weight 25})

          ;; Account al 30 dagen niet gebruikt? Slaap-verdacht.
          (and (:last-login user)
               (> (days-since (:last-login user) now) 30))
          (conj {:factor :dormant-account :weight 40}))

        score (reduce + (map :weight risk-factors))]

    {:risk-score    score
     :risk-factors  risk-factors
     :requires-mfa? (> score 50)}))
----

;; Lees dit. Je kent nu het volledige wachtwoordbeleid.
;; Geen Confluence-pagina. Geen Jira-ticket. Geen "vraag Sarah, zij heeft het geschreven."

(defn meets-password-policy? [password policy user-context]
  (let [violations
        (cond-> []
          (< (count password) (:min-length policy 8))
          (conj {:code :too-short  :message "Minimaal 8 tekens"})

          (and (:require-uppercase policy)
               (not (re-find #"[A-Z]" password)))
          (conj {:code :missing-uppercase  :message "Een hoofdletter vereist"})

          (and (:require-numbers policy)
               (not (re-find #"\d" password)))
          (conj {:code :missing-number  :message "Een cijfer vereist"})

          ;; Persoonlijk tintje: je eigen e-mailnaam mag niet je wachtwoord zijn
          (and user-context
               (str/includes? (str/lower-case password)
                              (str/lower-case (email-username (:email user-context)))))
          (conj {:code :contains-email  :message "Mag je e-mailadres niet bevatten"}))]

    {:valid? (empty? violations)  :violations violations}))
----

from datetime import datetime
from typing import Optional

@dataclass(frozen=True)
class LoginBeslissing:
    toegestaan: bool
    reden:      Optional[str] = None

                 config: dict,
                 nu: datetime) -> LoginBeslissing:
    if gebruiker_data is None:
        return LoginBeslissing(toegestaan=False, reden="Niet gevonden")
    if not gebruiker_data["active"]:
        return LoginBeslissing(toegestaan=False, reden="Gedeactiveerd")
    if gebruiker_data.get("lockout_until") and nu < gebruiker_data["lockout_until"]:
        return LoginBeslissing(toegestaan=False, reden="Vergrendeld")
    return LoginBeslissing(toegestaan=True)
----

=== Je Kunt Dit Vandaag Doen in Java/Spring

.Core — gewoon Java, nul Spring-imports
[source,java]

// core/UserRegistrationCore.java — geen @Service, geen @Autowired, niets
public final class UserRegistrationCore {

    // Onveranderlijk resultaattype — records zijn compact in Java 16+
    public record RegistratieBeslissing(boolean toegestaan, String reden) {
        public static RegistratieBeslissing doorgaan() {
            return new RegistratieBeslissing(true, null);
        }
        public static RegistratieBeslissing weigeren(String reden) {
            return new RegistratieBeslissing(false, reden);
        }
    }

    // Pure functie: heeft de shell een bestaande gebruiker gevonden? Wat moet er gebeuren?
    public static RegistratieBeslissing checkDuplicaat(String email, boolean bestaatAl) {
        if (bestaatAl) return RegistratieBeslissing.weigeren("E-mail al in gebruik");
        return RegistratieBeslissing.doorgaan();
    }

    // Pure functie: bouw het volledige gebruikersrecord — shell levert tijd en ID
    public static GebruikersRecord voorbereidAanmaken(String email, String passwordHash,
                                                      UUID id, Instant aangemaaktOp) {
        return new GebruikersRecord(id, email, passwordHash, aangemaaktOp, true, 0);
    }
}
----

.Shell — Spring-service, orkestreert I/O, roept core aan
[source,java]
----
@Service
public class UserShellService {

    private final UserRepository  userRepository;
    private final PasswordEncoder passwordEncoder;

    public UserShellService(UserRepository repo, PasswordEncoder encoder) {
        this.userRepository  = repo;
        this.passwordEncoder = encoder;
    }

    public User registreer(RegisterRequest req) {
        // Stap 1: shell doet DB-query, core neemt de beslissing
        boolean bestaatAl = userRepository.existsByEmail(req.email());
        var beslissing    = UserRegistrationCore.checkDuplicaat(req.email(), bestaatAl);
        if (!beslissing.toegestaan()) throw new EmailAlreadyExistsException(beslissing.reden());

        // Stap 2: shell doet I/O, core bereidt het record voor
        String hash    = passwordEncoder.encode(req.password());
        var prepared   = UserRegistrationCore.voorbereidAanmaken(
                             req.email(), hash, UUID.randomUUID(), Instant.now());

        // Stap 3: shell slaat op
        return userRepository.save(prepared.toEntity());
    }
}
----

* "Java-devs: je kent dit patroon al. Het is eigenlijk een statische utility-klasse met goede returntypes."
* "De `UserRepository`-interface is de port. Spring Data JPA is de adapter. Je schrijft het al zo — de stap is alleen zorgen dat je *bedrijfslogica* niet in de service lekt."
* "Records (Java 16+) maken onveranderlijke resultaattypes bijna zo compact als Clojure-maps."
* "`Instant.now()` verhuist naar de shell. Nu kun je elk tijdstempel meegeven aan `voorbereidAanmaken` in tests — geen `Mockito.mockStatic`."

== Probeer Het in Één Module

Kies *één ding* in je huidige codebase. Een model met echte bedrijfslogica. Een service die te veel doet.

. Maak `core/dat_ding.py` / `core/DatDingCore.java` / `core/dat_ding.rb`. **Geen framework-imports toegestaan.** Geen.
. Verplaats elke bedrijfsregel daarin als een gewone functie of statische methode.
. Laat de functies hun dependencies accepteren als argumenten — tijd, bestaande data, config.
. Schrijf tests voor die functies met *geen* mocks.
. Bouw een dunne service/shell bovenaan die de I/O doet en de pure functies aanroept.

Dat is het. Één module. Kijk hoe het voelt.
